# 워커 동시성 모델과 성능 특성 (스케일링 판단 근거)

- 일자: 2026-07-10
- 대상: `app/worker.py`(폴링 루프) · `app/handlers.py`(`_handle_classify` 이미지 루프) ·
  `app/core/deps.py`(`build_face_extractor`) · `app/pipeline/embed.py`(ORT 스레드) ·
  `app/pipeline/hdbscan_standalone.py`(N×N 거리 행렬)
- 목적: **워커를 몇 개 띄울지, 인스턴스 메모리를 얼마로 잡을지, 느릴 때 무엇부터 손댈지**를
  추측이 아니라 실측으로 판단한다.
- 관련: [ADR 007](../decisions/007-embedding-storage-s3.md) §4(동시 쓰기 직렬화 = SQS FIFO) ·
  [클러스터링 리뷰](../reviews/2026-07-09-clustering-review.md) 1번(O(N²) 규모 가드 부재) ·
  [임베딩 리뷰](../reviews/2026-07-09-face-embedding-review.md) 2번(ORT 스레드 하드코딩)

## 요약

1. **워커 1개 = 프로세스 1개**다. 워커 수를 정하는 설정값은 코드에 없다 — 배포 시 프로세스/컨테이너
   수로 정한다.
2. **사진은 한 장씩 순차 처리**된다(Python 스레드 1개). 다만 임베딩 **연산 내부**는 onnxruntime이
   C++ 스레드 8개로 병렬화한다. "순차"와 "멀티스레드"는 층위가 달라 둘 다 참이다.
3. **워커를 늘려도 같은 이벤트는 빨라지지 않는다.** FIFO `messageGroupId = event_id`가 이벤트 단위로
   직렬화하기 때문이다(ADR-007 §4). 워커 수는 **동시 진행 중인 서로 다른 이벤트 수**에 맞춘다.
4. **병목이 둘로 갈린다**: 시간은 임베딩(전체의 ~90%), 메모리는 클러스터링(O(N²)). 서로 다른 축이므로
   대응도 다르다.
5. MVP는 **4GB RAM · 워커 1개**로 충분하다. 단 이벤트당 얼굴 8,000개를 넘기면 OOM 위험이 실재한다.

## 1. 동시성 모델

### 1.1 프로세스 · 스레드 층위

워커 프로세스 안에 **Python이 아는 스레드**와 **Python이 모르는 스레드**가 따로 있다. 실측으로 세면:

| 시점 | Python 스레드 | OS 스레드 |
|---|---:|---:|
| 모델 로딩 전 | 1 | 1 |
| `FaceEmbedder()` 생성 후 | 1 | **10** |
| `embed_batch(32)` 실행 후 | 1 | 10 |

`FaceEmbedder()` 생성 시점에 onnxruntime이 C++ 내부에 스레드 풀(`intra_op_num_threads=8`)을 미리 띄운다
— 추론 시작 시 만드는 게 아니라 세션 생성 시 만들어 재사용한다.

```
워커 프로세스
│
├─ Python 층 ─── 스레드 1개
│   for ref in request.images:          # handlers.py:213 — 사진을 한 장씩 순차
│       image = fetch(...)              #   S3 대기 (CPU 유휴)
│       extracted = extract_faces(...)  #   session.run()에서 블로킹 (GIL 해제)
│
└─ onnxruntime C++ 층 ─── 스레드 8개
     행렬곱을 나눠 8코어에서 동시 계산
```

요리사 한 명이 주문을 하나씩 처리하되, 요리 하나를 만들 때는 화구 8구를 동시에 쓰는 구조다.

### 1.2 사진 단위 스레드를 두지 않은 이유

의도된 설계이며 근거가 코드에 있다.

- **`FaceDetector`가 스레드 안전하지 않다** (`detect.py:84`): `setInputSize()` + `detect()`가 내부 상태를
  공유하는 2단계 호출이라 동시 호출 시 레이스가 난다. `deps.py:47`이 "단일 스레드 워커 전제"로 이를 받는다.
- **코어가 이미 포화다**: `intra_op_num_threads`를 8→4로 줄이면 얼굴당 비용이 31ms→39ms로 **오른다**(§3.2).
  사진 단위 스레드를 얹으면 4×8 = 32스레드가 8코어를 다투는 오버서브스크립션이 된다
  (`embed.py:31` 주석이 경고하는 그 함정).
- **메모리가 배로 든다**: 스레드마다 `FaceDetector`를 따로 만들어야 하므로 모델도 스레드 수만큼 올라간다.

### 1.3 워커 여러 개는 안전하다 — 단, FIFO 덕분

`.npz`는 S3에 있고 S3에는 락이 없다. 워커의 쓰기는 `load → append → 재군집 → save`인 read-modify-write라,
같은 이벤트를 두 워커가 동시에 처리하면 lost update가 난다(삭제된 얼굴이 되살아남 = PIPA 재발).

이를 락이 아니라 **SQS FIFO로 막는다**. 인바운드 큐가 FIFO 하나이고 `messageGroupId = event_id`인데,
FIFO는 같은 그룹의 메시지를 **한 번에 하나씩만** 인도한다. 앞 메시지를 삭제하기 전까지 같은 이벤트의
다음 메시지는 어느 워커에게도 가지 않는다.

```
event-A 메시지 ──┐
                  ├─→ FIFO 큐 ─→ 워커 N대
event-B 메시지 ──┘

같은 event = 한 번에 한 대 (직렬)   |   다른 event = 서로 다른 워커 (병렬)
```

`face_id`가 `uuid4`(`handlers.py:189`)라 워커끼리 충돌하지도 않는다.

**깨지는 조건**: FIFO는 *같은 큐를 소비하는* writer만 직렬화한다. 큐를 거치지 않고 `.npz`를 건드리는
프로세스(주기 스윕 Lambda 등)를 추가하면 이 보호가 사라지고, S3 조건부 쓰기(If-Match ETag)가 필요해진다.
삭제 요청을 별도 Lambda로 빼지 않고 같은 워커에 물린 이유가 이것이다 (ADR-007 §3·§4).

## 2. 메모리 실측

### 2.1 모델 상주 · 추론 피크

```
인터프리터만            13 MB
+ numpy, cv2            56 MB
+ onnxruntime           68 MB
+ boto3                 86 MB
+ YuNet                107 MB
+ AuraFace             494 MB   ← glintr100.onnx 261MB가 대부분
+ 눈감음 CNN            495 MB   ← 유휴 상주
```

12MP(4032×3024) 사진 1장을 detect하고 얼굴 20개를 배치 임베딩할 때 피크 **865MB**. `_handle_classify`가
사진을 한 장씩 순회하므로 이미지 메모리는 누적되지 않는다.

**모델은 프로세스마다 각자 올라간다 — 공유되지 않는다.** 워커 2개 = 유휴 1GB.

### 2.2 클러스터링이 진짜 병목이다 — O(N²)

`_pairwise_distance`가 `np.asarray(X, dtype=np.float64)`로 **float64 승격** 후 N×N 행렬을 만들고,
`_mutual_reachability_graph`가 `np.partition` 사본과 `mr` 행렬을 더한다.

| 이벤트 내 얼굴 수 | N×N 1개(이론) | 실측 피크 | 행렬 환산 | 재군집 시간 |
|---:|---:|---:|---:|---:|
| 2,000 | 32 MB | 156 MB | 4.9배 | 0.10초 |
| 4,000 | 128 MB | 527 MB | 4.1배 | 0.33초 |
| 6,000 | 288 MB | 1,378 MB | 4.8배 | 0.59초 |
| 8,000 | 512 MB | 2,909 MB | 5.7배 | 1.04초 |
| 10,000 | 800 MB | **5,279 MB** | 6.6배 | 1.49초 |

> **클러스터링 리뷰 1번의 추정치 정정**: 해당 리뷰는 "N×N 행렬 2개 동시 보유 → N=1만에서 ~1.6GB"로
> 추정했으나, 실측은 **5.3GB**로 3배 이상 크다. 피크에 공존하는 N×N 등가물이 2개가 아니라 5~6개이며,
> 그 배수가 N이 커질수록 늘어난다(4.9배 → 6.6배). 규모 가드의 시급성은 리뷰의 판단보다 높다.

**"사진 수"가 아니라 "얼굴 수"다.** 단체사진 위주 이벤트는 사진 1,000장이 얼굴 5,000개가 된다.

### 2.3 4GB 예산 배분

| 항목 | 크기 |
|---|---:|
| OS + 컨테이너 오버헤드 | ~500 MB |
| 모델 상주 (워커 1개) | ~500 MB |
| 이미지 디코딩·추론 피크 | ~370 MB |
| **클러스터링에 남는 예산** | **~2.7 GB** |

→ 얼굴 **~5,000개까지 안전**, **~8,000개는 아슬아슬**, **10,000개 이상은 OOM**.

**OOM은 조용히 오지 않는다.** OOM killer가 프로세스를 죽이면 메시지는 미삭제 상태라 가시성 타임아웃 후
재전달되고, 같은 이벤트를 또 재군집하다 또 죽어 `max_receive_count`를 채우고 DLQ로 간다. 무한 루프는
아니지만 그 이벤트는 영영 처리되지 않는다.

## 3. 시간 실측

### 3.1 단계별 비용

| 단계 | 시간 |
|---|---:|
| YuNet detect (12MP) | 33 ms |
| YuNet detect (2MP) | 17 ms |
| AuraFace embed (얼굴 1) | **99 ms** |
| AuraFace embed (얼굴 5) | 204 ms |
| AuraFace embed (얼굴 20) | 712 ms |
| 눈감음 CNN (얼굴 1, 양눈) | 0.2 ms |
| `blur_variance` | ~0 ms |

**비용의 ~90%가 AuraFace 임베딩이다.** 품질 게이트는 사실상 공짜다. 클러스터링도 4,000 얼굴에 0.34초로
전체 대비 무시할 수준 — 시간 병목과 메모리 병목이 서로 다른 축이라는 점이 이 시스템의 핵심 특성이다.

얼굴 3명짜리 사진 1장 ≈ `33 + 156 + 1 ≈ 190ms` + S3 다운로드(100~300ms) = **장당 0.3~0.5초**.

| 사진 수 | 처리 시간 (장당 0.4초 가정) |
|---:|---:|
| 100장 | 40초 |
| 500장 | 3분 20초 |
| 1,000장 | 6분 40초 |

### 3.2 embed_batch에는 호출당 70ms의 고정비가 있다

배치 크기를 바꿔가며 선형 피팅하면:

```
embed_batch 1회 비용 ≈ 70ms (고정) + 31ms × 얼굴 수
```

| 배치 | 총 시간 | 얼굴당 |
|---:|---:|---:|
| 1 | 99 ms | 99 ms |
| 3 | 156 ms | 52 ms |
| 20 | 704 ms | 35 ms |
| 32 | 1,062 ms | 33 ms |
| 128 | 4,293 ms | 34 ms |

얼굴 1개짜리 사진은 99ms 중 **70ms가 순수 오버헤드**다. 사진 1,000장이면 고정비만 70초를 태운다.
배치 32 근처에서 얼굴당 33ms로 수렴하므로 그 이상 키워도 이득이 없다.

현재 `embed_batch`는 **사진 한 장 단위로 호출**된다(`deps.py:69`) — 한 장 안의 얼굴만 묶고, 사진끼리는
묶지 않는다. 즉 사진마다 70ms 고정비를 새로 낸다.

### 3.3 스레드를 더 줘도 빨라지지 않는다

| 설정 | 고정비 | 얼굴당 |
|---|---:|---:|
| 현재 (intra=8, inter=8) | 70 ms | **31 ms** |
| inter=1 | 69 ms | 31 ms |
| intra=4, inter=1 | 56 ms | 39 ms |
| intra=2, inter=1 | 40 ms | 64 ms |

`intra_op_num_threads`를 줄이면 얼굴당 비용이 그대로 배가 된다 = ORT가 이미 코어를 다 쓰고 있다.
`inter_op_num_threads`는 값을 1로 낮춰도 차이가 없다 — 임베딩 리뷰 2번의 "ORT_SEQUENTIAL에서는 죽은
설정"이라는 부수 발견이 실측으로 **확인**됐다.

**병렬성은 이미 짜내고 있다. 남은 건 배치뿐이다.**

## 4. 스케일링 판단 기준

### 4.1 워커 추가로 해결되는 문제 / 안 되는 문제

| 증상 | 워커 추가 | 올바른 대응 |
|---|---|---|
| 이벤트 하나가 느리다 | **효과 없음** (FIFO 직렬화) | vCPU 증설, 배치 개선, prefetch |
| 여러 이벤트가 동시에 몰린다 | 효과 있음 | 워커 추가 (+ 메모리 동반 증설) |
| 큰 이벤트에서 OOM | 효과 없음 (악화) | 메모리 증설, 규모 가드 |

워커를 10개 띄워도 사진 1,000장짜리 이벤트 하나는 여전히 한 워커가 6분 40초 동안 혼자 처리한다.
나머지 9개는 논다.

### 4.2 보아야 할 CloudWatch 지표

큐 깊이(`ApproximateNumberOfMessagesVisible`)가 아니라 **`ApproximateAgeOfOldestMessage`**를 본다.

- 나이가 계속 커진다 → 처리가 유입을 못 따라간다 → 스케일 판단 필요
- 깊이만 크고 나이는 짧다 → 그냥 이벤트 하나가 큰 것 → 워커를 늘려도 소용없다

이 둘을 구분해야 **워커를 늘릴지 vCPU를 늘릴지**가 갈린다. 여기에 **이벤트당 얼굴 수** 로그를 남겨
§2.2 표의 어디쯤에 있는지 확인한다(클러스터링 리뷰 2번 "재군집 요약 로그 부재"와 같은 항목).

### 4.3 인스턴스 사이징

- **MVP 권장**: 4GB · 워커 1개. 동시 진행 이벤트가 하루 두세 건이면 워커를 늘려도 논다.
- **워커 2개가 필요해지면 8GB로 간다.** 4GB에 워커 2개는 모델만 1GB이고, 두 워커가 동시에 큰 이벤트를
  재군집하면 즉시 터진다. **EC2를 여러 대 만들기 전에 한 대 안에서 프로세스를 늘리는 게 먼저다.**
- **vCPU와 스레드 설정을 정합시킨다.** ~~`DEFAULT_INTRA_OP_THREADS = 8`이 컨테이너 vCPU와 무관하게 항상
  8이다.~~ → **2026-07-11 해결**: `deps.py`가 가용 코어 수(`sched_getaffinity`)에 맞춰 `EmbedConfig`를
  주입한다(`Settings.ort_num_threads`로 오버라이드 가능, 0=자동). 이 경고가 실제 배포에서 **6배 손실**로
  터진 기록은 §7 참고.

## 5. 개선 후보 (우선순위)

| 개선 | 예상 이득 | 난이도 | 상태 |
|---|---|---|---|
| 1. 버릴 사진 임베딩 생략 | 흔들림·눈감음 비율만큼 (10~20%) | 작음 | **미착수 (신규 발견)** |
| 2. S3 다운로드 prefetch | 네트워크 대기 제거 (~30%) | 작음 | 미착수 |
| 3. 사진 가로지르는 배치(32) | 임베딩 단계 ~2배 | 중간 | 미착수 |
| 4. 클러스터링 규모 가드 | OOM 방지 | 중간 | 클러스터링 리뷰 1번 |
| 5. 워커 프로세스 추가 | **같은 이벤트엔 0** | — | 위 넷 이후 |

### 5.1 버릴 사진을 임베딩하고 있다 (신규 발견)

`deps.py:58-70`이 품질 판정 후 **무조건** 임베딩한다:

```python
eyes_closed, blurry = judge_faces(face_pairs, eye_classifier, quality_config)
...
embeddings = [e for e in embedder.embed_batch(crops) if e is not None]   # ← 항상 실행
return ExtractedFaces(embeddings, eyes_closed, blurry)
```

그런데 `handlers.py:230`이 그 결과를 폐기한다:

```python
if request.options.exclude_blurry and extracted.blurry:
    blurry_images.append(ref.image_id)
    continue          # ← 방금 계산한 embeddings를 통째로 버림
```

토글이 기본 ON이므로(feature-spec §6.1), 흔들리거나 눈감은 사진은 100~200ms를 들여 임베딩한 뒤 즉시
버려진다. `build_face_extractor`가 요청의 `options`를 모르기 때문에 생긴 구조다 — `extract_faces`에
토글을 넘겨 조기 반환하거나 `ExtractedFaces`를 지연 평가로 바꾸면 해결된다.

### 5.2 S3 다운로드 prefetch

`fetch`는 네트워크 대기라 CPU를 쓰지 않는다. 지금은 다운로드 중(100~300ms) CPU가 놀고, 임베딩 중
네트워크가 논다. **다운로드 전용 스레드 2~3개**만 두면 모델을 건드리지 않아(스레드 안전성 무관) 코어 경합
없이 대기를 없앨 수 있다. 사진 단위 스레드가 위험한 것과 달리 이쪽은 안전하다.

### 5.3 사진 가로지르는 배치

사진 1,000장 × 얼굴 2개(2,000 얼굴) 기준:

```
현재:    1,000회 호출 × (70 + 2×31)  = 132초
배치 32:    63회 호출 × (70 + 32×31) =  67초
```

두 패스(전체 detect·align → 32개씩 임베딩)로 바꾸면 된다. 정렬 crop은 얼굴당 37KB(112×112×3)라 2,000개를
모아도 74MB로 부담이 없다(임베딩 리뷰 "배치 메모리 상한 불요"와 정합).

### 5.4 클러스터링 메모리 완화

가장 싼 개선은 `_pairwise_distance`의 `float64` 승격을 `float32`로 낮추는 것이다(메모리 절반). 임베딩이
애초에 float32이고 cosine 거리는 `[0,2]` 범위라 정밀도는 충분하나, **ADR 005가 "PoC 라벨 완전 일치"를
계약으로 걸어둔 터라 라벨 동등성 재검증이 선행돼야 한다.** 코드에 이미 `out=` 인자로 N×N 사본을 없앤
최적화 흔적(`hdbscan_standalone.py:106`, `:129`)이 있어, 이 경로가 메모리 민감하다는 점은 인지된 상태다.

## 7. EC2 실측 — 스레드/코어 정합이 6배를 가른다 (2026-07-11)

첫 EC2 배포(t4g.small · **2 vCPU** arm64/Graviton2, Spring과 동거) 후 "로컬보다 느리다"는 관찰에서 출발한
측정이다. §4.3이 경고했던 오버서브스크립션이 실제로 터진 사례이며, **페널티가 예상보다 훨씬 컸다**.

### 7.1 스레드 수별 임베딩 비용 (동일 워크로드, 얼굴 30개)

| intra/inter | 로컬 (15코어) | **EC2 (2코어)** |
|---:|---:|---:|
| 8 | **63.8 ms/장** | **2,860 ms/장** ← 배포됐던 값 |
| 4 | 72.6 ms/장 | 1,225 ms/장 |
| 2 | 127.5 ms/장 | **476 ms/장** ← 최적 |
| 1 | 224.8 ms/장 | 815 ms/장 |

12MP 얼굴 감지: 로컬 25ms · EC2 269ms.

**핵심: 코어 수를 넘는 스레드는 이득이 없는 게 아니라 급격한 손해다.** 2코어에 8스레드를 주면 최적(2스레드)
대비 **6배** 느려진다. §3.3이 "스레드를 줄이면 느려진다"고 관찰한 것은 **코어가 충분할 때만** 성립하는
국소적 결론이었다 — 코어가 모자라면 부호가 뒤집힌다.

부차적으로, 스레드가 코어보다 적을 때도 손해다(EC2 1스레드 815ms > 2스레드 476ms). **정답은 코어 수와
같게 두는 것**이고, 그래서 `deps.py`가 `sched_getaffinity`로 코어를 세어 주입한다.

### 7.2 스로틀링은 원인이 아니었다

t4g는 버스터블이라 CPU 크레딧 소진을 먼저 의심했으나 **아니었다**: `CPUCreditBalance`가 576(만점)에서
한 번도 감소하지 않았고 `CPUUtilization`은 최고 12%였다. 느린 이유는 스로틀링이 아니라 **스레드가 코어를
서로 뺏느라 CPU를 제대로 못 쓰고 있었기 때문**이다(그래서 사용률도 낮게 나온다).

### 7.3 남은 격차는 하드웨어다

수정 후에도 EC2는 로컬 대비 **7.5배** 느리다(476ms vs 63.8ms). 이건 코어 수(2 vs 15)와 코어당 성능
(Graviton2 vs Apple M) 차이라 설정으로 못 줄인다. 더 빠르게 하려면 **vCPU를 늘리는 것**이 유일한 지렛대다
(§4.1의 "이벤트 하나가 느리다 → vCPU 증설"과 정합 — 워커를 늘려도 FIFO 때문에 소용없다).
