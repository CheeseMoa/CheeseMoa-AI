# ADR 003: 인물 군집은 전체 재군집을 원천으로 한다

## Status

Accepted

## Context

이벤트(event)에 사진이 추가될 때 인물 클러스터를 어떻게 갱신할지 두 방식을 비교했다.
(인물 앨범은 모임 안의 이벤트 단위로 만들어지므로 재군집 격리 단위는 event다 — [ADR 007](./007-embedding-storage-s3.md).)

- **(A) 증분 대표벡터 매칭**: 기존 클러스터를 대표벡터(centroid)로 요약해두고, 신규 임베딩만
  HDBSCAN으로 묶어 대표벡터를 만든 뒤 기존 대표벡터와 코사인 임계값으로 병합한다.
- **(B) 전체 재군집**: 개별 임베딩을 전부 보관하고, 매 트리거마다 event 전체 임베딩(기존+신규)을
  HDBSCAN으로 다시 군집화한다.

서비스 우선순위는 **분류 정확도**다. 인물 오분류는 사용자 경험에 직접 영향을 준다.

## Decision

**(B) 전체 재군집을 군집의 진실로 삼는다.**

1. 개별 얼굴 임베딩을 **전량 보관**한다(재군집의 전제). 저장 substrate는 event 단위 S3 `.npz`
   ([ADR 007](./007-embedding-storage-s3.md)) — 이 ADR의 원칙은 저장소 선택과 무관하게 유지된다.
2. 매 트리거마다 event 전체 임베딩을 HDBSCAN으로 **전체 재군집**한다.
3. 재군집 파티션을 기존 클러스터와 **overlap 최대 매칭(Jaccard / 헝가리안)**으로 연결해
   `cluster_id`를 승계한다(업로드 간 인물 번호 연속성 유지).
4. 사용자 보정(merge/split/reassign)은 **must-link / cannot-link 제약**으로 반영해
   재군집이 사람의 결정을 뒤집지 않게 한다.
5. 대표벡터(L2 정규화 평균)는 **조회·표시용 파생 캐시**로 강등한다. 군집 판단의 근거가 아니다.

## Rationale

- **정확도**: 대표벡터는 사람 얼굴의 다봉(multi-modal) 분포를 1점으로 뭉개고, 병합 단계가
  HDBSCAN의 밀도 판단이 아닌 nearest-centroid 휴리스틱이 된다. 초기 오류가 고착·누적되며,
  작은 신규 배치에서 밀도 기반 군집이 특히 불안정하다. sklearn `HDBSCAN`은 네이티브 증분
  (`approximate_predict`)도 없다. 전체 재군집은 데이터가 쌓일수록 스스로 교정한다.
- **비용**: 클러스터링은 전체 파이프라인 연산의 0.1% 미만이다([ADR 002](./002-hdbscan-sklearn.md)).
  비싼 단계는 이미지당 감지·임베딩이지 군집이 아니므로, 이벤트당 수백 벡터 전체 재군집은 부담이 없다.
- **연속성**: 전체 재군집은 파티션이 실행마다 바뀔 수 있으나, overlap 재조정으로 `cluster_id`를
  승계해 정확도와 인물 번호 안정성을 동시에 확보한다.
- **보정 보존**: 사람의 보정을 제약으로 걸어 재군집이 이를 훼손하지 않는다.

## Consequences

- 저장소([ADR 007](./007-embedding-storage-s3.md))는 클러스터 요약이 아니라 **개별 임베딩 전량**을
  보관해야 한다(요약본만으론 재군집 불가).
- `app/pipeline/cluster.py`는 HDBSCAN 재군집 + `cluster_id` 재조정 로직을 함께 담당한다.
- 결과 메시지의 `is_new`는 "이번 재군집에서 기존 `cluster_id` 매칭에 실패한 신규 인물"을 뜻한다.
- **규모 확장 시**: 재군집 격리 단위가 event(이벤트당 수백 벡터)라 규모 부담은 사실상 없다. 다만
  극단적으로 이벤트가 수만+로 커지면, 증분 근사 매칭을 fast-path로 쓰고 **주기적 전체 재군집**으로
  드리프트를 교정하는 탈출구가 있다. 이때도 정답 기준은 전체 재군집이다. (feature-spec §10 #1)
- HDBSCAN 파라미터(`min_cluster_size`·`eps`)와 재조정 overlap 임계는 데이터 특성에 따라
  재조정이 필요하다(feature-spec §10 #4).
