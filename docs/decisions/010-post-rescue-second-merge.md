# ADR 010: 2차 파편 병합 — 구제·축출 뒤 최종 멤버십에서 병합 재판정

## Status

Accepted (2026-07-14). [ADR 005](./005-hdbscan-standalone-port.md)의 파편 병합,
[ADR 008](./008-blob-promotion-connected-components.md)·[ADR 009](./009-clustering-parameter-tuning.md)의
임계 체계 위에 후처리 단계 하나를 추가한다. 임계값 자체는 전부 현행 유지.

## Context

실서비스 event(16번)에서 같은 인물이 인물 앨범 2개로 남는 사례가 보고됐다. 대표벡터로 재현·진단한 결과:

- 파편 병합(`_merge_fragments`)은 **구제·축출 전** centroid 스냅샷으로 판정한다. 해당 event에서 두 파편의
  판정 시점 유사도는 **0.688** — 임계(`merge_centroid_similarity=0.7`)에 0.012 미달로 병합이 안 됐다.
- 직후 노이즈 구제가 한 파편에 멤버를 추가하면서 centroid가 이동, **최종 구성 기준 유사도는 0.705**로
  임계를 넘었다. 즉 "파이프라인 스스로 동일 인물로 정의한 수준(≥0.7)의 쌍"이 판정 시점 차이로 남는다.
- 같은 event에서 완전 연결 가드에 막혔던 다른 파편 쌍도 최종 기준으로는 0.740이었다(같은 인물).
- cannot-link 제약·중복 임베딩(P0 오염)은 해당 event에 없음을 확인 — 순수한 판정 시점 문제다.

## Decision

`recluster()`의 후처리 마지막(저신뢰 축출 직후, ID 재조정 전)에 **같은 임계·같은 cannot-link 가드로
`_merge_fragments`를 1회 더 적용**한다 (2차 파편 병합, 모듈 독스트링 ⑥).

## 검증 (실 데이터 전수 + 합성 회귀)

- **S3 실 event 17개 전수 시뮬레이션**: 2차 병합으로 결과가 달라진 event는 문제의 16번 **하나뿐**.
  두 파편 쌍(0.705, 0.740)이 각각 병합되어 인물 앨범 4개 → 정답인 2개가 됐고, **오병합 0건**
  (미병합 쌍 최대 유사도 0.28). 사용자 보정 제약이 있는 event(4·13·14 등) 포함 나머지 전부 결과 불변.
- **저해상도 스트레스(스크린샷 9장, 2인)**: 파편 유사도가 0.689로 임계 미달인 채 유지 — 발동 조건
  미성립 시 결과가 완전히 동일함(무해성)을 확인.
- `cluster.py` 합성 자가검증 **(j)** 로 회귀 고정: 1차 판정 0.69(미달) → 구제 편입 후 ≈0.707 → 2차 병합
  발동 기하를 닫힌형식 벡터로 재현.

## Rationale

1. **추가 위험이 없다.** 같은 임계의 재적용이라 "동일 인물 수준"의 정의를 바꾸지 않고, 완전 연결·
   cannot-link 가드도 그대로다. 발동 조건(후처리가 경계선 클러스터의 멤버를 실제로 바꿈)이 좁아
   현상 유지가 기본값이다.
2. **병합을 구제 뒤로 "이동"하는 대안은 기각.** 현행 순서 주석대로 노이즈 구제는 병합된(큰) centroid
   기준으로 도는 것이 의도라, 이동하면 구제가 파편난 작은 centroid 기준이 되어 구제율이 떨어질 수 있다.
   이동이 아니라 추가(1차는 유지)로 시점 문제만 보완한다.
3. **오병합보다 미병합(ADR-006) 원칙과 양립.** 임계 하향(ADR-008·009가 기각한 0.65 등)과 달리 타인
   대역(실측 상한 0.635)을 전혀 침범하지 않는다.

## Consequences

- 반복 1회 추가일 뿐 수렴 루프는 아니다 — 2차 병합 후 다시 구제/축출을 돌리지 않는다(멤버십 변화가
  다시 경계선을 만들 수는 있으나, 실 데이터 전수에서 필요 사례가 없었고 루프는 결정성·비용 부담).
  재발 시 반복 횟수 확장을 검토한다.
- 병합으로 흡수된 파편의 기존 `cluster_id`는 ID 재조정에서 자연스럽게 은퇴(`retired_cluster_ids`)로
  흘러 Spring에 통지된다 — 계약 변경 없음.
