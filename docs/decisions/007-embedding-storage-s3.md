# ADR 007: 임베딩 저장소는 S3 blob(event 단위 `.npz`)을 쓴다

## Status

Accepted — [ADR 003](./003-full-reclustering.md)이 저장소로 지정한 pgvector를 대체한다.
ADR 003의 **"전체 재군집이 군집의 진실"** 원칙 자체는 유지되며, 이 ADR은 *임베딩을 어디에
어떤 단위로 두느냐*(저장 substrate와 재군집 격리 단위)만 바꾼다.

## Context

AI 워커의 raw face embedding을 어디에 저장할지 재결정이 필요했다. ADR 003은 pgvector를
전제했으나 다음이 드러났다.

- **pgvector의 핵심 기능(ANN 검색, HNSW/IVFFlat)이 이 워크로드에서 놀고 있다.** ADR 003대로
  매 트리거마다 해당 단위의 전체 임베딩을 numpy로 통짜 로드해 HDBSCAN을 다시 돌리므로,
  "이 벡터의 최근접 K개" 같은 벡터 인덱스 질의를 **한 번도 하지 않는다.** 인덱스는 죽은 무게다.
- **pgvector = PostgreSQL 확장**이라, 쓰려면 Postgres 인스턴스를 상시 띄워야 한다. RDS든 ECS든
  24시간 요금이 발생하는데, 이 서버는 **버스트 워크로드**(메시지 올 때만 처리)라 상시 인스턴스는 낭비다.
- **재군집 격리 단위가 event로 확정**됐다(인물 앨범은 모임 안의 이벤트 안에 만들어진다). 이벤트당
  얼굴 수는 **수백 규모(N≈56~300)**라 벡터 전용 인덱스의 이점이 없다. (scale justifies simplicity)

### 재군집 격리 단위: group → event

인물 앨범(= `cluster_id` = 앱의 `personId`)은 **이벤트(촬영 단위)** 안에서 만들어진다. 따라서
군집의 진실은 "그 **이벤트** 전체 임베딩(기존+신규)에 대한 HDBSCAN 재군집"이다. `group`(모임)은
멤버십·공유·접근 제어의 상위 컨테이너일 뿐, 재군집 격리 단위가 아니다.

- **결과**: 같은 사람이 서로 다른 이벤트에 찍히면 이벤트마다 별개 인물 앨범이 된다(이벤트 간 병합 없음).
  이것이 제품 설계다.
- 이 결정은 ADR 003·feature-spec의 "group 전체 재군집" 서술을 **event 전체 재군집**으로 정정한다.

## 핵심 요구사항

1. 임베딩 저장 (개별 벡터 유사도 검색 불필요, 이벤트 단위 통짜 로드)
2. 개별 사진/face 삭제 반영 가능 (소프트/하드)
3. 상시 인스턴스 요금 회피 (버스트 패턴)
4. 기존 아키텍처 경계 유지 (백엔드 ↔ AI 워커는 SQS 메시지로만 소통, 데이터 소유권 분리)
5. `recluster()`의 세 입력(임베딩 + 직전 `cluster_id` + 보정 제약)이 모두 저장될 자리

## 검토한 대안

| 방안 | 상시 요금 | 개별 삭제 | 비고 |
|---|---|---|---|
| RDS Postgres(pgvector) 상시 | 높음 (24h) | SQL DELETE | ANN 미사용이라 과함, 피하려는 대상 |
| Aurora Serverless v2 | 최소 ACU 잔존 (완전 0 아님) | SQL DELETE | 관계형이 꼭 필요할 때만 |
| DynamoDB 온디맨드 | 0 (인스턴스 시간 없음) | DeleteItem 즉시 | 즉시 물리 삭제 필요 시. 통짜 로드엔 부적합 |
| **S3 `.npz` + 지연 삭제** | **거의 0 (스토리지만)** | 재군집/삭제 메시지 시 처리 | **버스트·통짜 로드 패턴에 최적** |

## Decision

**event 단위 S3 파일(`.npz`) 저장 + 지연 삭제**를 채택한다. 별도 DB 없음, 상시 요금 없음,
아키텍처 경계 유지. 하드 삭제가 "요청 즉시"가 아니라 "합리적 기간 내(수일)" 처리로 충분하다는 전제다.

### 저장 레이아웃 — 한 이벤트 = 한 `.npz` 객체

```
s3://{bucket}/embeddings/{eventId}.npz
  embeddings        (N×D) float32   — ①  recluster() 입력
  face_ids          (N)              — id 매핑
  photo_ids         (N)              — id 매핑 + 삭제 마스킹 키
  cluster_ids       (N)              — ②  직전 cluster_id (previous_cluster_ids). 없으면 ID 승계 불가
  must_link_pairs   (K×2)            — ③  보정 제약 (face_id 참조)
  cannot_link_pairs (M×2)            — ③  보정 제약 (face_id 참조)
```

- **`recluster()`의 세 입력이 한 객체에 모두 있다.** 제약·직전 배정은 **face_id 참조(안정적)**로
  저장하고, `recluster()` 호출 직전에 **현재 행 인덱스로 번역**한다(행 인덱스는 매 재군집마다 바뀜).
- **한 파일에 다 넣는 이유**: (a) 한 S3 객체 = 한 번의 원자적 쓰기 → 임베딩/제약/삭제가 따로 놀아
  생기는 정합성 창이 없다. (b) 하드 삭제로 face가 사라지면 그 face를 참조하는 제약 쌍도 dangling이
  되는데, 어차피 파일 전체 rewrite니까 같은 마스크 로직에서 함께 정리된다.
- `.npy`는 단일 배열만 담으므로 다중 배열용 `.npz` 사용. 향후 컬럼 필터가 필요하면 parquet 전환 고려.

### 재군집 흐름 (분류 요청 수신)

```python
# 1. 메시지에서 eventId, deletedPhotos 추출
# 2. S3에서 {eventId}.npz 로드 (boto3) — 없으면 최초 군집
# 3. 새 사진 임베딩 추출 후 append (photo_id 기준 멱등 — 재처리 시 중복 append 금지)
# 4. 삭제 마스킹 적용 (아래 마스크 분리 참고)
# 5. face_id 기준 cluster_ids·제약을 현재 행 인덱스로 번역 → recluster() 실행
# 6. 정리된 배열을 S3에 다시 write → 여기서 하드 삭제가 물리적으로 반영됨
# 7. 결과 메시지(classify-result) 발행
```

핵심: 재군집 시 어차피 파일 전체를 rewrite하므로, 하드 삭제가 재계산에 공짜로 얹힌다.

## 반드시 챙길 4가지

### 1. 파일에 id 배열 동봉
임베딩만으론 어떤 벡터가 어떤 face/photo/cluster인지 모른다. `face_ids`·`photo_ids`·`cluster_ids`를
같은 파일에 저장한다. 특히 `cluster_ids`가 빠지면 `recluster()`가 직전 배정을 못 받아
[매번 전부 `is_new`가 되고 인물 번호 연속성이 깨진다](../spec/feature-spec.md#L143).

### 2. 삭제 마스크 — MVP는 단일 마스크 (복원 없음)
**결정(2026-07-03): MVP는 소프트 삭제 복원(restore) UI가 없다.** 소프트든 하드든 되살아나지 않으므로
AI가 삭제된 벡터를 파일에 남길 이유가 없다 — **소프트·하드 구분 없이 단일 제거 마스크**로 물리 제거한다
(생체정보 최소 보관에도 유리).

```python
# 복원 없음 → 파일 rewrite와 클러스터링 입력이 같은 마스크
keep_mask = ~np.isin(photo_ids, deleted_photo_ids)
```

- rewrite 시 삭제된 face_id를 참조하는 `must_link_pairs`·`cannot_link_pairs`도 함께 프루닝한다
  (dangling 제약 방지).
- **향후 복원 UI가 생기면** 이중 마스크로 되돌린다: `write_mask`(하드만 물리 제거) +
  `cluster_mask`(소프트·하드 모두 클러스터링 제외, 소프트는 파일 유지). 이 지점이 분기 재도입점이다.

### 3. 재군집 안 걸리는 이벤트의 PIPA 사각지대 차단 — 워커가 삭제 메시지 처리 (Lambda 불필요)
"하드 삭제 요청이 왔는데 재업로드가 없으면 재군집이 안 걸려 벡터가 방치된다"는 우려는 **워커가 업로드
때만 도는 경우**에만 성립한다. 워커는 **상시 폴링하는 장기 실행 프로세스**(feature-spec §3)이므로,
Spring이 **삭제 요청을 FIFO 큐로 발행**하면(분류·보정과 동일) 워커가 업로드와 무관하게 그 메시지를 받아
`{eventId}.npz`를 rewrite한다. **삭제 메시지 자체가 트리거**라 재업로드를 기다릴 필요가 없어 사각지대가 없다.

- **별도 Lambda는 두지 않는다(결정 2026-07-03).** 삭제 요청은 분류·보정과 똑같이 Spring이 큐로 발행하고
  같은 워커가 소비한다. 삭제는 모델이 필요 없어(numpy 마스크 + 쓰기뿐) 워커에 부담이 없다. Lambda를 두면
  워커와 함께 두 writer가 되어 단일 writer 불변식(4번)만 깨진다.
- (선택, post-MVP) 메시지 유실·보존기간 초과 대비 **주기 조정 스윕**이 필요하면 그때 스케줄 작업을 추가한다
  — 이는 per-request 삭제가 아니라 컴플라이언스 백스톱이다.

### 4. 동시 쓰기 직렬화 = SQS FIFO (단일 writer)
`{eventId}.npz`를 건드리는 경로가 여럿(분류·보정·삭제)이라 S3(락 없음)에서 lost update가 나면 삭제가
되살아난다(PIPA 재발). **SQS FIFO로 직렬화한다(결정 2026-07-03)**:

- 분류·보정·삭제 메시지를 **하나의 FIFO 큐**에 `messageGroupId = event_id`로 넣는다. FIFO는 한 그룹의
  메시지를 **한 번에 하나씩**만 인도하므로, 같은 이벤트의 쓰기가 겹치지 않는다.
- **워커가 그 이벤트 파일의 유일한 writer**다 — 분류·보정·삭제를 모두 이 워커가 처리해 rewrite한다
  (삭제는 모델 불필요, numpy 마스크 + 쓰기뿐). 별도 Lambda writer가 없어 단일 writer가 자연히 성립한다(3번).
- FIFO는 **같은 큐를 소비하는** writer만 직렬화한다 — 이것이 단일 writer를 유지하는 이유다. 훗날
  독립적으로 S3를 쓰는 프로세스(예: 주기 스윕)가 필요해지면 그때 S3 조건부 쓰기(If-Match ETag)를 얹는다.

## 폐기된 방안: AI 워커 → 백엔드 DB 직접 접근

검토했으나 기각:
- **소유권 경계 붕괴**: `face` 테이블을 백엔드/AI 양쪽이 write → 보호 로직에 race condition.
- **스키마 결합**: AI가 백엔드 테이블 구조에 직접 의존 → 컬럼 변경 시 조용히 깨짐. 계약 표면이
  메시지 스키마에서 테이블 전체로 확장.
- **SQS 비동기 모델과 충돌**: 백엔드 트랜잭션과 타이밍이 얽혀 write 순서 문제.

## Consequences

- pgvector·Postgres 인스턴스를 두지 않는다. 상시 요금 0, 스토리지 요금만.
- 저장소 접근은 인터페이스로 추상화한다(`recluster()`는 저장소를 모르는 순수 함수 유지) — S3 구현과
  단위 테스트용 인메모리 fake를 교체 가능하게.
- SQL의 트랜잭션·임의 쿼리·멀티라이터 안전성을 잃고 **동시성 제어를 손으로 떠안는다**(위 4번).
  event 단위 N이 작고(수백) 접근 패턴이 통짜 로드라 이 교환은 남는 장사다.
- 대표벡터(L2 정규화 평균)는 재군집 때마다 재계산되는 **파생값**이라 저장이 필수가 아니다 —
  결과 메시지로 Spring에 보내 표시용으로만 쓴다(ADR 003의 "대표벡터는 파생 캐시" 유지).
- event 단위라 N이 작아, ADR 003·005·feature-spec의 "모임당 수만/수십만 벡터" 규모 우려와
  새벽 배치 fast-path(§10 #1)는 **사실상 무의미**해진다.

## 결정됨 (2026-07-03)

- **소프트 삭제 복원 UI 없음(MVP)** → 소프트·하드 구분 없는 단일 제거 마스크(§반드시 챙길 2).
- **PIPA 삭제 = 워커가 처리(별도 Lambda 불필요)**(§3) — Spring이 삭제 요청을 FIFO 큐로 발행하고, 상시 폴링 워커가 소비해 `.npz` rewrite. 삭제 메시지 자체가 트리거라 사각지대 없음.
- **동시성 = SQS FIFO** `messageGroupId = event_id`, **워커가 이벤트 파일의 단일 writer**(§4).

## 미해결 / 확인 필요

- 하드 삭제 즉시성 요구 수준 재확인 (현재 "수일 내 허용" 가정, 생체정보라 보수적 검토 권장).
- 삭제 메시지 네이밍 (feature-spec §10 #7 SQS 큐 네이밍 TBD와 함께).
