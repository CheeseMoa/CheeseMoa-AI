# ADR 013: 배경 인물 얼굴 크기 필터 — rel_w 2.5% 하한

## Status

Accepted (2026-07-15). [ADR 012](./012-merge-threshold-recalibration.md)와 독립적인 신규 결정 —
파편화가 아니라 **원치 않는 앨범 생성**(배경 행인) 문제를 다룬다.

## Context

event 27(실사진 26장) 분류 결과에서 사진 2장짜리 앨범이 생겼는데, 주인은 두 사진의 배경에 멀리
찍힌 **지나가는 행인**이었다(44×55px, 47×62px — 이미지 긴 변의 0.77~0.82%). 같은 자리에서 연속
촬영한 두 장에 같은 배경 인물이 들어갔고, 두 임베딩이 서로 최근접(상호 유사도 0.477)이면서 다른
모든 얼굴과 동떨어져 `min_cluster_size=2`를 정확히 충족 — HDBSCAN이 2인 클러스터로 승격시켰다.
시스템 관점에서는 정상 동작이지만, 사용자에게는 모르는 사람의 앨범이다.

행인은 **진짜 얼굴**이라 score 필터로는 거를 수 없다(YuNet score는 높다). 남는 신호는 크기다.

### 분포 측정 (2026-07-15)

S3의 실사진 이벤트 16개 전체(4·8·12·13·14·16·17·19~27)를 검출 단계부터 재실행해, 각 얼굴을
event `.npz` 행과 임베딩 대조로 매칭 → 클러스터 배정 여부를 라벨링했다. 이벤트 간 재업로드
중복은 md5로 제거(325 → 210 얼굴).

| 그룹 | n | rel_w(bbox 폭/이미지 긴 변) 분포 |
|---|---|---|
| 앨범 배정 (지켜야 함) | 87 | **min 3.29%** · p5 4.12% · 중앙값 13.9% · max 42.9% |
| 행인 (걸러야 함) | 2 | 0.77% ~ **0.82%** |
| 노이즈 (미배정) | 121 | min 0.46% · 중앙값 2.04% · max 41.2% |

- 행인 최대 0.82%와 앨범 최소 3.29% 사이는 **앨범 얼굴이 0개인 빈 구간** — 4배 간격.
- 하한 스윕: **1.0~3.0% 전 구간이 앨범 손실 0 + 행인 2/2 제거**(고원). 3.5%부터 앨범 손실 시작.
- 경계의 앨범 얼굴은 육안 확인 — 3.29%(event 27 셀카 주인공), 3.76%(event 13 저해상도 단체사진
  아이) 둘 다 반드시 지켜야 하는 실제 주인공이다.
- 부수 효과: 하한 2.5%에서 노이즈 얼굴 121개 중 63개가 검출 단계에서 제거된다(임베딩 비용 절감).

## Decision

`DetectorConfig.min_face_rel_width` 기본값 **0.025** — bbox 폭이 이미지 긴 변의 2.5% 미만인
얼굴을 검출 단계에서 버린다. `.env` `DETECT_MIN_FACE_REL_WIDTH`로 조정, 0 = 비활성.

- 2.5%는 고원(1.0~3.0%)의 상단 쪽 — 배경 노이즈 제거를 늘리는 방향의 운영 선택이다. 행인 쪽
  마진은 3.0배로 두텁고, 앨범 최소값(3.29%) 쪽 마진은 **1.3배로 얇다** — 앨범에서 사진이
  누락된다는 리포트가 오면 이 값부터 내려볼 것(측정상 1.0%까지는 행인 차단이 유지된다).
- 적용 지점은 **검출 단계**(`detect()`, refine 이전): 걸린 얼굴은 임베딩·품질 게이트·재군집에
  아예 진입하지 않는다. 눈감음 게이트의 "얼굴 1개라도 해당" 정책이 배경 얼굴에 오염되는 것도
  함께 막는다.

## 기각한 대안

- **절대 픽셀 하한** (예: 검출 스케일 30px): 저해상도 업로드를 자른다 — event 13의 앨범 얼굴은
  558×418 이미지에서 21px(rel_w 3.76%)로, 절대 기준이면 소실된다. 상대 기준만 해상도 불변.
- **클러스터 승격 단계 필터** (작은 얼굴은 새 클러스터를 못 만들되 기존 앨범 구제는 허용):
  더 정교하지만 크기 정보를 `.npz`까지 운반해야 하고(스키마 확장), 노이즈 구제가 저품질 임베딩을
  끌어들이는 경로는 그대로 남는다. rel_w 1.5% 미만은 정렬 crop(112px)이 업스케일되는 수준이라
  임베딩 신뢰도 자체가 낮아, 구제 기회를 잃는 손실이 사실상 없다 — 단순한 검출 단계 필터로 충분.
- **score/종횡비 필터**: 행인은 진짜 얼굴이라 score가 높다. 별건인 화장품 팔레트 오검출
  ([분포 측정 리뷰](../reviews/2026-07-15-distance-distribution-verdict.md) §별건)에는 여전히
  후보로 유효하나, 이 문제와는 다른 지렛대다.

## 한계 · 리스크

- **행인 표본이 2개(1명, 1개 이벤트)뿐**이다. 앨범/노이즈 분포(208개)는 두터우나 "걸러야 하는
  쪽" 경계는 얇다 — 실 운영에서 행인 앨범 리포트가 다시 오면 그 케이스의 rel_w로 이 ADR을
  재검증할 것.
- **가까운 행인(rel_w > 3%)은 통과한다** — 이 필터는 "멀리 배경에 찍힌 얼굴"만 거른다. 카메라
  가까이 지나간 행인이 2장 이상 반복되면 여전히 앨범이 생길 수 있다.
- 하한에 걸린 얼굴은 임베딩이 없으므로 사용자 보정(reassign 등)의 대상도 될 수 없다 — 뒷줄에
  아주 작게 찍힌 일행의 사진은 그 사람 앨범에 안 들어간다. 완화: 임계 이하 얼굴은 임베딩
  신뢰도가 낮아 어차피 오분류 위험이 크고, `.env`로 즉시 하향/비활성 가능.

## Consequences

- event 27 행인 앨범 재현 시나리오가 기본값만으로 사라진다(회귀 검증: 행인 bbox 2개 소멸,
  같은 사진의 주인공·경계 앨범 얼굴 전부 유지).
- 검출 → 임베딩으로 넘어가는 얼굴 수가 줄어(측정 데이터 기준 노이즈 46% 제거) 대량 이벤트의
  추론 비용이 감소한다.
- 이후 크기 관련 튜닝(팔레트 오검출의 크기/종횡비 필터 등)은 이 측정 데이터
  (스크래치패드 `face_sizes.csv` 생성 스크립트: 이벤트 전체 재검출 + npz 임베딩 대조)를
  재실행해 같은 방법으로 근거를 만들 것.
