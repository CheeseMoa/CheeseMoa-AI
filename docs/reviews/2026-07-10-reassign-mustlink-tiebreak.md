# 결함 재현 기록 — reassign의 must-link 동률이 목적지 앨범을 쪼갠다

- 일자: 2026-07-10
- 대상: `app/pipeline/cluster.py`의 `_enforce_must_link`(동률 규칙) — 파급은 `_enforce_cannot_link`
  → `_merge_fragments` → `_match_cluster_ids`로 이어진다. 제약을 만드는 쪽은 `app/handlers.py`의
  `_translate_feedback` reassign 분기
- 상태: **수정 반영** (2026-07-10) — §6 수정 후보 (A) 채택: `_enforce_must_link`가 다수결 후보에서
  cannot-link 금지 라벨을 제외하도록 수정하고, §2 기하를 `cluster.py` `__main__` 자가검증 (i)로
  회귀 고정했다. §2 코드의 출력 주석은 **수정 전** 재현 결과이며, 수정 후에는
  `b (0, 3, 4)` / `a (1, 2)`(신규 앨범 없음)가 나온다. 반영 커밋: `36627d6`.
  §6의 "함께 필요한 것" 중 handlers 자가검증 ⑤ 강화와 §7 남은 확인 과제는 미해결로 남아 있다.
- 관련: [사용자 보정 리뷰](./2026-07-09-user-feedback-review.md) — 그 문서의 **"확인해서 문제 없었던 것"**
  두 항목("대표 얼굴 선정의 결정성"의 *대표 하나로 충분* 근거, "자가검증 커버리지"의 ⑤ reassign)을
  본 문서가 정정한다. 재군집 후처리의 순서·불변식 맥락은 [클러스터링 리뷰](./2026-07-09-clustering-review.md) 참조

## 1. 증상

사용자가 사진 **한 장**을 앨범 a → b로 옮기면, 앨범 b가 쪼개지고 **없던 앨범이 새로 생긴다.**
옮긴 사진은 목적지 앨범이 아니라 그 신규 앨범에 들어간다.

```
기대: 앨범 a = {p2, p3}          앨범 b = {p1, p4, p5}
실제: 앨범 a = {p2, p3}          앨범 b = {p5}          앨범 NEW = {p1, p4}
```

`p4`는 사용자가 건드리지도 않은 사진인데 원래 앨범에서 뽑혀 나온다. 그리고 이 결함은
`cluster_feedback`의 4종 액션 중 **reassign에만** 나타난다 — merge·split·confirm_distinct는
must-link 컴포넌트가 3명 이상이거나 must-link 자체가 없어 아래의 동률 조건을 밟지 않는다.

## 2. 재현

`.venv`와 numpy만 있으면 된다(모델·AWS 불필요). 목표 코사인 행렬에서 단위벡터를 역산해
실측 유사도 대역(동일 인물 0.46~0.70, 타인 ≲0.3)을 정확히 통제한다.

```python
import numpy as np
from app.pipeline.cluster import EMBED_DIM, Constraints, recluster

# f1 = B 사람의 애매한 사진(A로 오분류됨), f2·f3 = A 사람, f4·f5 = B 사람
g = np.eye(5)
g[0, 1] = g[1, 0] = g[0, 2] = g[2, 0] = 0.55   # f1 ↔ A  (오분류의 원인)
g[0, 3] = g[3, 0] = g[0, 4] = g[4, 0] = 0.45   # f1 ↔ B
g[1, 2] = g[2, 1] = 0.65                        # A 응집
g[3, 4] = g[4, 3] = 0.65                        # B 응집
for i in (1, 2):
  for j in (3, 4):
    g[i, j] = g[j, i] = 0.30                    # 타인

w, v = np.linalg.eigh(g)                        # PSD → 단위벡터 5개로 인수분해
emb = np.zeros((5, EMBED_DIM))
emb[:, :5] = v @ np.diag(np.sqrt(np.clip(w, 0.0, None)))
emb = (emb / np.linalg.norm(emb, axis=1, keepdims=True)).astype(np.float32)

# handlers._translate_feedback의 reassign 분기가 만드는 제약과 동일:
#   must   = (이동 얼굴 f1, 목적지 대표 f4)
#   cannot = (이동 얼굴 f1, 출처 잔류 대표 f2)
counter = iter(f"NEW-{k}" for k in range(1, 9))
result = recluster(
  emb,
  ["a", "a", "a", "b", "b"],
  constraints=Constraints(must_link=((0, 3),), cannot_link=((0, 1),)),
  new_id_factory=lambda: next(counter),
)
for c in result.clusters:
  print(c.cluster_id, c.member_indices, "신규" if c.is_new else "")
# NEW-1 (0, 3) 신규      ← f1·f4
# a     (1, 2)
# b     (4,)             ← f5만 남음
```

B 사람의 응집도를 0.50으로 낮춘 변형에서도 동일하게 실패한다.

## 3. 원인

내부 라벨의 이동을 단계별로 찍으면 사고 지점이 ②다.

```
① HDBSCAN 원본     L0:{f1, f2, f3} | L1:{f4, f5}     ← 예상대로 f1을 또 A에 넣는다
② must-link 강제    L0:{f1, f2, f3, f4} | L1:{f5}     ← 목적지 대표 f4가 B에서 끌려나온다
③ cannot-link 강제  L0:{f1, f4} | L2:{f2, f3} | L1:{f5}
④ 파편 병합         (변화 없음 — centroid 0.646 < 0.7)
```

**② `_enforce_must_link`의 동률 규칙.** reassign의 must-link 컴포넌트는 `{이동 얼굴, 목적지 대표}`
단 2명이라 비노이즈 다수결이 **항상 1대1 동률**이다. 동률에서 채택하는 "작은 라벨"은 HDBSCAN이
내부적으로 매긴 번호일 뿐 출처/목적지와 아무 의미 관계가 없다. 출처(A)가 작은 번호를 받으면
목적지 대표가 출처 클러스터로 끌려간다.

여기서 채택된 라벨(A)은 **cannot-link가 이미 금지한 라벨**이다 — `f1`은 `f2`와 cannot-link이고
`f2`의 라벨이 곧 A다. 즉 ②는 ③이 반드시 되돌릴 선택을 하고 있다.

**③ 되돌리는 과정이 앨범을 부순다.** `_enforce_cannot_link`는 위반 라벨 L0을 앵커 `{f1, f4}`(2명)와
`{f2}`(1명)로 가르는데, **큰 앵커가 원 라벨을 유지**하는 규칙 때문에 `{f1, f4}`가 L0을 차지하고
정작 A 사람인 `f2`·`f3`가 새 라벨로 밀려난다. `f5`는 ②에서 혼자 남겨진 채다.

**④ 마지막 구제 실패.** `{f1, f4}`와 `{f5}`는 같은 사람이지만 centroid 유사도가 **0.646**으로
병합 임계 `merge_centroid_similarity = 0.7`에 미달한다. 실측 동일 인물 centroid가 0.6~0.7 대역
([ADR 008](../decisions/008-blob-promotion-connected-components.md))이므로 이 실패는 예외가 아니라
흔한 경우로 봐야 한다.

**⑧ Jaccard 매칭이 신규 id를 발급한다.** 겹침이 큰 짝부터 배정되어 `{f2,f3}`가 a(0.667),
`{f5}`가 b(0.5)를 가져가고, `{f1,f4}`는 a(0.25)·b(0.333) 양쪽 모두 선점당해 **승계 실패 → 신규 앨범**.

### 두 실험으로 좁힌 근거

| 실험 | 결과 | 함의 |
|------|------|------|
| 행 순서만 뒤집어 B가 라벨 0을 받게 함 | ✅ 정상 동작 | 결함이 **HDBSCAN 라벨 번호라는 우연**에 좌우된다 |
| 다수결 후보에서 cannot-link 금지 라벨 제외 | ✅ 두 순서 모두 정상 | ②가 단일 원인임이 확정 |
| (대안) 목적지 **전 얼굴**과 must-link | ✅ 정상 | 동률이 사라져서 해결되나 부작용 있음(§6) |

## 4. 왜 기존 자가검증이 못 잡았나

`handlers.py`의 자가검증 ⑤가 reassign을 덮고 있고 29건 전부 통과한다. 단언이 이렇다:

```python
image_ids_of(reassigned, containing="img-a1").isdisjoint({"img-a3"})
and "img-a3" in image_ids_of(reassigned, containing="img-b1")
```

"옮긴 사진이 출처와 떨어졌나" + "옮긴 사진이 목적지 대표와 같은 클러스터인가". **결함이 발현된
상태에서도 두 단언이 모두 참이다** — 옮긴 얼굴과 목적지 대표가 나란히 신규 앨범에 들어가기 때문이다.

빠진 단언은 셋이다: 목적지 **cluster_id가 승계**되었는가, 목적지의 **나머지 멤버가 유지**되는가,
**신규 앨범이 생기지 않았는가**. 대표 얼굴 하나만 확인하는 단언은 "앵커 쌍이 함께 움직이는가"만
보므로, 앵커가 앨범을 끌고 나오는 이 실패 모드에 구조적으로 눈이 멀어 있다.

## 5. 발현 조건 (언제 실제로 터지는가)

세 조건이 모두 성립해야 한다.

1. **HDBSCAN이 옮긴 사진을 출처 앨범에 다시 넣는다.** 임베딩은 바뀌지 않았으므로 오분류는 반복된다
   — reassign이 필요했던 이유 그 자체라 **흔하다**.
2. **출처 클러스터의 HDBSCAN 라벨 번호가 목적지보다 작다.** 데이터·행 순서에 따라 갈린다.
   결정적이지만 예측 불가능하고, 의미적 근거가 없다.
3. **파편 병합이 목적지 조각을 다시 붙이지 못한다.** 목적지 사람의 centroid 유사도 < 0.7.
   실측 대역상 드물지 않다.

역으로 2번이 반대로 나오면 아무 일도 없이 정상 동작하므로, **같은 조작이 이벤트마다 성공하거나
실패하는 비결정적으로 보이는 증상**으로 관측될 수 있다(실제로는 입력에 대해 결정적이다).
운영에서 "옮겼는데 앨범이 이상해졌어요" 문의가 재현되지 않는 형태로 들어올 수 있다.

## 6. 수정 후보

### (A) 권장 — `_enforce_must_link`를 cannot-link 인지형으로

다수결 후보에서 **컴포넌트 멤버의 cannot-link 상대가 현재 가진 라벨**을 제외한다. 전부 금지되면
지금처럼 새 라벨을 발급한다. 필요한 정보(`constraints.cannot_link`)는 이미 `recluster`가 쥐고 있다.

근거가 이 모듈의 기존 설계와 정확히 같은 계열이다 — `rescue_similarity ≥ min_membership_similarity`,
`blob_promote_floor ≥ min_membership_similarity` 두 불변식은 모두 **"한 단계가 한 일을 다음 단계가
곧바로 되돌리는 churn"** 을 생성 시점에 막는 규칙이다. ②가 ③이 되돌릴 라벨을 고르는 것도 같은 churn이고,
차이는 되돌리는 과정에서 앨범 구조가 파손된다는 점뿐이다.

검증 결과(작업 트리에 미반영, 시험 적용 후 원복):

| 검증 | 결과 |
|------|------|
| `python -m app.pipeline.cluster` 합성 자가검증 | 13건 전부 통과 |
| `python -m app.handlers` 스모크 검증 | 29건 전부 통과 |
| 본 문서 §2 재현 시나리오 2종 | 둘 다 정상 |

### (B) 비권장 — `handlers.py`가 목적지 전 얼굴과 must-link

동률이 사라져 증상은 없어지지만, 제약 개수가 목적지 앨범 크기에 비례해 늘고(`.npz` 영속),
**알고리즘이 묶은 앨범 구성을 사용자 결정으로 굳혀버린다.** must-link는 "사용자가 같다고 확정한
관계"라는 의미를 잃는다. 게다가 [사용자 보정 리뷰 §1](./2026-07-09-user-feedback-review.md)의
앵커 생존 의존 문제가 제약 수만큼 증폭된다.

### 함께 필요한 것 — 자가검증 보강

수정 여부와 무관하게 ⑤의 단언을 강화해야 한다. 최소한:

- 목적지 `cluster_id`가 승계될 것 (`is_new == False`)
- 목적지의 기존 멤버가 전부 목적지에 남을 것
- 결과에 신규 앨범이 생기지 않을 것

현재 `fake_extractor`의 `img-a3`는 A 사람 얼굴이라 §2의 기하를 그대로 재현하지 못할 수 있다.
회귀 고정에는 **"오분류된 B 사람 사진"** 에 해당하는 얼굴 벡터를 fake에 추가해야 한다
(⑬ confirm_distinct가 실측 오병합 기하를 합성으로 고정한 것과 같은 방식).

## 7. 재현의 한계 (해석 시 주의)

이 절이 본 문서의 신뢰 구간이다.

- **합성 임베딩이다.** 레포에 얼굴 이미지가 없어 실사진 AuraFace 임베딩으로는 재현하지 못했다.
  §2는 5×5 코사인 행렬을 고유분해해 만든 벡터로, **지정한 쌍 유사도는 정확하지만** 실제 512차원
  임베딩의 분포·국소 밀도·MST 구조를 대표하지는 않는다. HDBSCAN의 condensed tree가 실사진에서
  같은 라벨링을 낼지는 별도 확인이 필요하다.
- **유사도 값은 실측 대역에서 골랐다.** 동일 인물 0.46~0.70, 타인 ≲0.3은 `cluster.py`
  주석과 [ADR 008](../decisions/008-blob-promotion-connected-components.md)에 기록된 face-test 실측치다.
  다만 "A로 오분류될 만큼 A에 가까운 B 사람 사진"(f1↔A 0.55 > f1↔B 0.45)의 **빈도**는 측정된 바 없다.
  결함의 존재는 확인됐지만 **발생률은 미지수**다.
- **§5의 2번 조건(라벨 번호 대소)을 확률로 말할 수 없다.** HDBSCAN 라벨 부여는 입력에 대해
  결정적이며 무작위가 아니다. "행 순서를 뒤집으면 뒤집힌다"는 것만 실험으로 보였을 뿐,
  실데이터에서 어느 쪽으로 치우치는지는 조사하지 않았다.
- **수정 후보 (A)의 검증은 기존 자가검증 42건 통과에 근거한다.** 그 검증들이 §4에서 드러났듯
  이 실패 모드에 눈이 멀어 있으므로, **"기존 검증 통과"는 무회귀의 강한 증거가 아니다.**
  §6의 자가검증 보강이 선행되거나 최소한 동반되어야 한다.
- **다른 액션으로의 파급은 검사만 하고 재현하지 않았다.** merge·split은 컴포넌트가 3명 이상이라
  동률을 밟지 않고, `__uncertain__` reassign은 이동 얼굴이 노이즈(-1)라 다수결이 목적지 라벨로
  일의적으로 결정된다 — 코드 독해에 근거한 판단이며 실행으로 확인하지 않았다.

### 남은 확인 과제

1. 실사진으로 §2 기하를 재현 — `python -m app.pipeline.cluster <이미지들>` CLI가 이미 있다.
   "A와 헷갈리는 B 사람 사진"을 포함한 5장이면 된다.
2. face-test 데이터에서 f1↔출처 > f1↔목적지인 오분류 사진의 비율 측정 → §5의 1번 조건 정량화.
3. `_match_cluster_ids`의 greedy Jaccard가 §3 ⑧처럼 **작은 조각에 신규 id를 발급**하는 다른 경로가
   있는지 점검 — 본 결함은 ②가 원인이지만, 신규 앨범 발급이라는 최종 증상은 매칭 단계에서 난다.
4. **제3앨범 동률 (이론상 잔존 구멍, 2026-07-10 분석)** — 수정 (A)는 금지 라벨(출처)만 제외할 뿐
   "작은 라벨" 동률 규칙 자체는 남아 있다. 앨범이 3개 이상이고 이동 사진이 재군집에서 **출처도
   목적지도 아닌 제3앨범 클러스터**에 흡수되면, 컴포넌트 `{이동 얼굴, 목적지 대표}`의 투표가
   제3앨범 1표 vs 목적지 1표 동률이 되고(제3앨범은 어떤 cannot-link도 금지하지 않음), 제3앨범의
   라벨 번호가 작으면 목적지 대표가 그쪽으로 끌려가 §1과 같은 목적지 파손이 재발한다.
   - 재현 시도: 앨범 3개(C·A·B) 기하 계열에서 C 크기·응집도·행 순서 약 90개 조합 스캔 —
     "이동 얼굴의 제3앨범 흡수"까지는 발현되나 "제3앨범 라벨 < 목적지 라벨"이 한 번도 나오지 않아
     **끝까지 발현시키지 못했다.**
   - 배제도 불가: 최종 라벨은 condensed tree 노드 id의 정렬 순서다
     (`hdbscan_standalone.py`의 `cluster_map = {c: n for n, c in enumerate(sorted(clusters))}`) —
     덴드로그램 구조의 산물이라 행 순서·앨범 의미와 무관하고, 목적지가 항상 작은 라벨을 받는다는
     구조적 보장이 없다. 스캔 실패는 해당 기하 계열의 트리 구조 우연으로 봐야 한다.
   - 참고: merge의 동률(예: 2장+2장)은 무해하다 — 컴포넌트가 양쪽 전원이라 어느 라벨이 이기든
     구성이 같고, 최종 id는 Jaccard가 내용 기준으로 승계한다. 동률이 위험한 건 컴포넌트가 작아
     투표 결과에 따라 누군가 자기 클러스터에서 뽑혀 나오는 reassign 패턴뿐이다.
   - 해법 후보: 동률을 "작은 라벨" 대신 의미 기준으로 — (가) 컴포넌트 평균 벡터와 후보 클러스터
     centroid의 유사도 비교(목적지 후보엔 대표 자신이 포함돼 닻 역할, 함수 내 재료로 충분),
     (나) `previous_cluster_ids`를 넘겨 "이전에도 그 앨범이던 멤버"가 있는 후보 우선. (가) 권장.
     merge류 대칭 동률에서는 다시 동률이 나 기존 규칙으로 떨어지지만 그 경로는 무해하다.

## 부록 — 이 문서가 정정하는 선행 판단

[2026-07-09 사용자 보정 리뷰](./2026-07-09-user-feedback-review.md)의 "확인해서 문제 없었던 것"
두 항목을 무효화한다.

| 선행 판단 | 정정 |
|-----------|------|
| **대표 얼굴 선정의 결정성** — "대표 하나로 충분한 이유(`_enforce_cannot_link`가 위반 라벨을 쪼갠 뒤 나머지를 최근접 앵커로 재배정)도 주석에 명시" | 대표 하나로 **충분하지 않다.** 그 주석은 `_enforce_cannot_link`만 보고 있으며, 그 앞의 `_enforce_must_link` 동률이 대표를 엉뚱한 라벨로 끌고 간다는 사실을 놓쳤다. 대표 선정의 *결정성*은 여전히 옳다(행 순서 고정) — 결함은 결정성이 아니라 **동률 해소 규칙의 의미 부재**다 |
| **자가검증 커버리지** — "기하를 이기는 사용자 결정(⑤)" | ⑤는 앵커 쌍의 동거만 단언해 목적지 앨범의 파손을 통과시킨다 (§4) |

선행 리뷰가 틀렸다기보다, **정적 독해로는 ②③④⑧의 상호작용이 보이지 않았다.** 이 결함은 네 단계를
모두 통과시켜야 증상이 드러나며, 단계별 라벨을 찍어보는 실행 추적으로만 잡힌다. 후속 리뷰에서
다단계 후처리 파이프라인을 볼 때의 교훈으로 남긴다.
