# 코드 리뷰 — 정렬 크롭 → 임베딩 경로 (유지보수성 · 운영 관점)

- 일자: 2026-07-09
- 대상: `app/pipeline/embed.py`(AuraFace 임베딩) + 정렬→임베딩 접합부 `app/core/deps.py`의
  `extract_faces`, 그리고 임베딩이 영속되는 `app/storage/event_embeddings.py`와의 상호작용
- 관점: 유지보수성, 운영 환경(실 AWS 배포)에서 발생 가능한 문제
- 상태: **발견 기록 — 전 항목 미수정** (수정 시 각 항목에 반영 커밋을 기입할 것)
- 관련: [감지 리뷰](./2026-07-09-face-detection-review.md) ·
  [정렬 리뷰](./2026-07-09-face-alignment-review.md) — 공통 발견(모델 revision 미고정,
  관측성 부재, config↔Settings 미연결 패턴, 조용한 드롭)은 그 문서들을 따르고 여기서는
  **임베딩 단계에 고유한 발견**만 다룬다.

## 요약

임베딩 코드 품질은 높다 — PoC 레시피의 수치 동일 이식, 배치=단건 동등성 검증 스크립트 존재,
오류 전략(프로그래밍 오류 raise vs 데이터 퇴화 None)의 명시적 구분이 모두 잘 되어 있다.
고유 리스크는 하나가 크다: **임베딩은 S3 `.npz`에 영속·누적되는데 어떤 모델이 만든 벡터인지
기록되지 않는다** — 모델이 바뀌면 한 event 안에 비교 불가능한 두 임베딩 공간이 섞여 군집
품질이 조용히 붕괴하고, 원본 전량 재처리 없이는 복구할 수 없다. 다른 파이프라인 단계와 달리
임베딩은 산출물이 **저장소에 남는** 단계라서, 모델 드리프트의 피해가 일회성이 아니라 영속이다.

## 운영 환경 리스크

### 1. 모델 출처(provenance)가 `.npz`에 미기록 — 임베딩 공간 혼합 리스크

`event_embeddings.py`의 `.npz`에는 `schema_version`(레이아웃 버전)만 있고 **어떤 모델·버전이
임베딩을 생산했는지**에 대한 기록이 없다. 임베딩은 event 단위로 영속·누적되며 매 재군집의
입력 원천이다(ADR-003·007).

운영 영향: AuraFace 모델 파일이 바뀌면 — HF revision 미고정(감지 리뷰 2번)으로 인한 의도치
않은 업스트림 갱신이든, 향후 의도적 모델 업그레이드든 — 그 시점 이후 추가된 얼굴만 새 공간의
벡터로 append된다. 서로 다른 모델의 임베딩 공간은 코사인 비교가 무의미하므로, 같은 `.npz` 안에
두 공간이 섞이는 순간 HDBSCAN 재군집이 event 단위로 조용히 망가진다. 증상은 "어느 날부터 새
사진이 자꾸 새 인물로 갈라진다"류의 품질 저하로만 나타나고, **저장된 벡터만으로는 원인 판별이
불가능**하며, 복구는 event의 원본 사진 전량 재임베딩뿐이다.

참고로 ORT 버전·CPU 마이크로아키텍처 차이로 인한 수치 미세 드리프트는 코사인 허용 오차 안에
있어 무해하다 — 치명적인 것은 모델 파일 교체다.

- 권장: ① `.npz`에 모델 식별자(모델 파일 sha256 또는 고정 revision 문자열) 필드 추가 —
  `schema_version`이 이미 있으므로 v2 마이그레이션 경로로 자연스럽다, ② 로드 시 현재 임베더의
  모델 식별자와 불일치하면 명확히 표면화(StoreCorruptionError와 구분되는 별도 신호 권장 —
  손상이 아니라 마이그레이션 필요 상태), ③ 모델 업그레이드 runbook(전량 재임베딩 절차) 설계.
  감지 리뷰 2번(revision 핀)이 선행되면 발생 확률 자체가 크게 줄지만, 기록이 없으면 의도적
  업그레이드 때 같은 문제를 만난다.

### 2. ORT 스레드 수 8 하드코딩 (PoC 8vCPU 기준) + `EmbedConfig` Settings 미연결

`DEFAULT_INTRA_OP_THREADS = 8`은 PoC의 배포 타깃(8vCPU) 검증값인데, `deps.py`가
`FaceEmbedder()` 기본값으로 생성하므로 실제 컨테이너 vCPU와 무관하게 항상 8이다. 1~2 vCPU
컨테이너면 오버서브스크립션(스로틀링·지연 널뛰기), 16 vCPU면 미활용이다. 감지 리뷰
1번(`cv2.setNumThreads` 부재)·6번(config↔Settings 미연결)과 같은 계열로, **배포 시 cv2 스레드·
ORT 스레드·컨테이너 CPU limit 세 값을 한 곳(워커 부트스트랩 + Settings)에서 정합**시켜야 한다.

부수 발견: `inter_op_num_threads`는 ORT 기본 실행 모드(ORT_SEQUENTIAL)에서는 효과가 없는
설정이다(병렬 실행 모드 전용). 현재 8로 선언되어 "PoC가 검증한 의미 있는 튜닝값"처럼 읽히지만
실제로는 죽은 설정일 가능성이 높다 — 확인 후 제거하거나 근거 주석을 정정할 것.

- 권장: `Settings.to_embed_config()` 추가 + 배포 컨테이너 스펙 확정 시(배포 로드맵 항목)
  vCPU에 맞춰 주입. `EmbedConfig` 주석이 이미 이 시나리오("워커 부트스트랩에서 주입")를
  예고하고 있으므로 구조 변경 없이 값 배선만 하면 된다

### 3. `log_severity_level = 3` — 세션 전체 WARNING 억제

알려진 무해 경고 1종(출력 메타데이터 `{1, 512}` 고정 선언으로 인한 배치 shape 경고)을 지우기
위해 ERROR 미만 로그를 전부 끈다. 근거는 주석으로 문서화되어 있고 로그 오염 방지 자체는
타당하지만, 프로바이더 폴백·메모리 경고 등 **운영에서 유용했을 다른 WARNING도 함께 사라진다**.
ORT는 경고를 종류별로 끄는 수단이 없어 현실적 대안이 마땅치 않은 트레이드오프다.

- 권장: 현 상태 유지 가능. 단, 운영 장애 조사 시 "ORT 경고는 억제되어 있다"를 인지할 것 —
  주석에 이미 있으므로 추가 조치는 관측성 로그(감지 리뷰 4번) 도입 시 재평가만

## 유지보수성

### 4. 소소한 것들

- **출력 차원 검증의 사각**: `FaceEmbedder.__init__`의 `EMBED_DIM` 검사는 출력 shape 메타데이터가
  정수일 때만 동작한다(`isinstance(output_dim, int)`). 심볼릭 차원으로 선언된 잘못된 모델은
  로딩 시점을 통과하고, `EventEmbeddings.__post_init__`의 shape 검증에서 뒤늦게 잡힌다 —
  잡히긴 하므로 우선순위 낮음. 로딩 직후 더미 1건 추론으로 실제 출력 shape을 확정하는 방법이 있다.
- **onnxruntime `>=1.26,<2.0` 범위 핀**: 감지 리뷰 7번(opencv 상한)과 동일 유형. 임베딩이
  영속된다는 점에서(1번) 재현성 요구가 다른 의존성보다 높으므로, 배포 시 정확 핀을 권장.
- `embed.py` `__main__` 스모크와 `scripts/verify_embed_equivalence.py`의 pytest 승격은
  로드맵(pytest 도입)에 편승 — 특히 동등성 스크립트는 배치 경로의 유일한 안전망이므로 승격 우선.

## 확인해서 문제 없었던 것 (오탐 방지 기록)

- **배치=단건 동등성 주장의 근거**: 주석이 언급하는 "동등성 검증 스크립트"가
  `scripts/verify_embed_equivalence.py`로 실제 레포에 존재한다 — 주장이 재현 가능하다
  (감지 리뷰 1번·정렬 리뷰 2번의 "문서가 존재하지 않는 것을 가리킴" 유형이 아님).
- **접합부 인덱스 정합**: `deps.py`의 `extract_faces`에서 품질 판정(`judge_faces`)은 정렬
  실패(None) 크롭을 안전하게 건너뛰고(`aligned is not None` 가드 확인), 임베딩 입력은 None
  필터 후 순서를 보존해 `embed_batch`로 들어간다 — 얼굴 단위 identity가 필요 없는 계약
  (face_id는 핸들러가 임베딩 시점에 발급)이라 어긋날 인덱스 자체가 없다.
- **오류 전략의 명시적 구분**: `_preprocess`는 shape/dtype 위반을 프로그래밍 오류로 보고
  ValueError로 크게 실패(→ 핸들러의 이미지 단위 격리로 `failed_images` 수렴), `_l2_normalize`는
  데이터 퇴화를 None으로 격리 — 구분 기준이 독스트링에 있고 detect/align의 None 패턴과 일관된다.
- **PoC 의도적 이탈의 문서화**: 영벡터를 PoC처럼 그대로 반환하지 않고 None으로 거르는 이유
  (코사인 거리 미정의 → HDBSCAN·대표벡터 보호)가 근거와 함께 명시되어 있다.
- **`CPUExecutionProvider` 명시 고정**: GPU 빌드가 섞여 설치돼도 PoC 검증 환경과 동일 동작 —
  재현성 관점에서 올바른 결정.
- **배치 메모리**: (N, 3, 112, 112) float32는 얼굴 50개 기준 약 7.5MB — 단체 사진 워크로드에서도
  무시 가능한 수준, 배치 크기 상한 불요.
- **잘못된 모델 주입의 조기 검출**: 출력 차원 검사를 첫 추론이 아니라 로딩 시점에 수행
  (`AURAFACE_MODEL_PATH` 오설정 방어) — 위 4번의 사각을 제외하면 올바른 설계.
- **`EMBED_DIM` 3중 선언**(embed/cluster/event_embeddings): storage→pipeline 의존 금지라는
  근거가 각 선언부에 문서화되어 있어 의도된 중복이다.

## 우선순위

| 순위 | 항목 | 비용 |
|------|------|------|
