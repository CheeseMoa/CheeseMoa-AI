# 자연어 사진 검색 — 이미지·텍스트 임베딩 모델 조사 (2026-07-20)

향후 "말로 사진 찾기"(예: 한국어로 "물컵", "동상", "건물", "노을 진 바다"를 입력하면 해당 사진을
찾음) 기능을 붙이기 위한 모델 조사다. **판정: 상용 가능·arm CPU·한국어를 동시에 만족하는 사실상
유일한 후보는 SigLIP 2 ViT-B(Google, Apache 2.0)다.** 아직 착수 결정은 아니며, 채택 전 실측
선행 과제가 남아 있다.

이 조사는 두 개의 별개 질문에서 출발했다: ① 인형·그림·팔레트 같은 비-실물 얼굴 오검출을
거르는 방법(anti-spoofing 조사, 같은 날 별도 진행), ② 자연어 사진 검색. 사용자의 실제 목표는
②이며, 검색용 이미지 임베딩이 ①의 필터로도 겸용될 수 있는지가 부수 관심사다(아래 미해결로 남음).

## 왜 "객체 탐지기"가 아니라 "이미지·텍스트 임베딩"인가

자연어 검색은 물체마다 박스를 치는 탐지기(YOLO 등)가 아니라 **CLIP 계열 공동 임베딩**으로 푼다.
업로드 시 사진 1장당 임베딩 벡터 1개를 뽑아 저장하고(지금 AuraFace 얼굴 임베딩을 event `.npz`에
쌓는 것과 동일 패턴), 검색 시 질의 텍스트를 같은 공간의 벡터로 바꿔 코사인 유사도로 랭킹한다.
즉 현행 "임베딩 저장 + 벡터 검색" 인프라에 그대로 얹힌다. 참고로 객체 탐지 경로의 대표 격인
Ultralytics YOLOv5/v8/v11은 **AGPL-3.0**이라 소스 비공개 상용 서비스엔 부적합 —
AuraFace를 non-commercial 때문에 배제한 것과 같은 이유로 기본 제외.

### 1순위 — SigLIP 2 ViT-B 근거

- **라이선스**: `timm/ViT-B-16-SigLIP2` 및 공식 `google/siglip2-*` 릴리스가 Apache 2.0. 팀이 늘
  걸리던 "코드만 오픈, 가중치 research-only" 함정이 여기선 없다(가중치까지 permissive).
- **다국어**: 논문 제목이 "SigLIP 2: Multilingual Vision-Language Encoders". WebLI(109개 언어)로
  학습, Gemma 토크나이저(256k vocab)가 한글 처리. 다국어 검색(XM3600)이 원조 대비 22.5%→40.7%,
  전용 다국어판 mSigLIP에 거의 근접하면서 영어는 더 강함.
- **성능**: 원조 SigLIP를 전 사이즈에서 능가. ImageNet zero-shot B/16@256 79.1%(vs 76.7%),
  COCO text→image R@1 47.4%→53.2%. 증류 덕에 작은 B 모델에서 향상 폭이 커 저사양에 유리.
- **통합**: ONNX 변환 가능 → AuraFace와 동일한 "벡터 1개 저장 후 검색" 패턴 그대로.
- 4개 사이즈(ViT-B 86M / L 303M / So400m 400M / g 1B) 중 **ViT-B가 최소**라 CPU/엣지 최적.

### 배제 사유 요약

- **MobileCLIP**: 기술적으론 최적(SigLIP ViT-B보다 2.3배 빠르고 정확, ONNX 준비됨 — 단 2.3배는
  Apple 기기 실측이지 arm t4g 아님). 그러나 가중치가 Apple Research License(비상용), 데이터가
  CC-BY-NC-ND, **파생 모델까지** 상용 금지. AuraFace 회피와 정확히 같은 함정이라 결정적 탈락.
- **KoCLIP**: OpenAI CLIP의 한국어 포팅(text=klue/roberta-large). 코드 Apache지만 가중치가
  MS COCO 2014 + AI Hub 한국어 캡션 파인튜닝이라 상용 적법성 미확인.
- **jina-clip-v2**: 865M(≈3.4GB fp32 / int8도 865MB+)로 1.8GB 호스트(Spring 공유)에 과중 +
  CC-BY-NC-4.0.
- **MetaCLIP**: 코드 MIT라 폴백으로 남길 가치는 있으나 가중치·데이터 라이선스는 이번에 미확인.

## 남은 선행 과제 (채택 전 실측·확인 필수)

이 조사는 문헌·라이선스 확인까지이며, 아래는 실측/서면 확인이 필요한 열린 항목이다.

1. **SigLIP 2 라이선스 잔여 리스크** — 가중치는 Apache 2.0이나 **WebLI는 구글 비공개 사유
   데이터셋**. "Apache 승인이 WebLI 파생 가중치까지 면책하는가"를 AuraFace 실사 기준 그대로
   서면 확인하는 것이 이상적(승인 문구 자체는 permissive).
2. **한국어 실측 부재** — 다국어 강점은 109개 언어 집계치뿐, **한국어 단독 검색 품질 수치 없음**.
   "물컵/동상/노을 진 바다" 실제 질의로 직접 검증, 안 되면 "한국어→영어 번역 후 임베딩" 우회책과
   A/B 비교.
3. **arm t4g 실비용 미측정** — SigLIP 2 ViT-B의 onnxruntime CPU 지연(ms/장)·RAM을 t4g.small에서
   잰 수치 없음. 양자화/배칭 필요 가능성. 사진 1장당 1회(업로드 시)라 얼굴만큼 빈번하진 않음.
4. **겸용(인형/동상 필터) 미확정** — "같은 SigLIP 2 임베딩을 인형·그림·동상 zero-shot 필터로도
   재활용" 가능성은 이번 검증 주장에 안 잡혔다. **가설로만 두고**, 검색 프로토타입 때 함께 실측.
   (별개 anti-spoofing 조사 결론: 2D 인쇄·그림은 MiniFASNet(Apache)로 걸러지나, 3D 인형·마네킹은
   원리적으로 잔여 오검출이 남음 — 밀랍인형 벤치마크 최고 성능도 ACER 11.73%.)
5. **커버 못 한 각도** — OpenCLIP/EVA-CLIP/TinyCLIP/M-CLIP, RAM·Tag2Text 등 태그 기반 검색은
   이번 검증 주장에 미포함. 임베딩 방식이 자연어 자유 질의에 우위라는 일반론까지만 확인됨.

## 아키텍처 메모

- 저장: 사진 1장당 임베딩 벡터 1개(차원은 모델 확정 후) — event `.npz` 패턴 재사용 가능.
- 검색: 소규모에선 brute-force 코사인으로 충분, 규모 커지면 인덱스 도입 검토(별도 결정).
- 모델 소싱: SigLIP 2가 HF Hub 배포이므로 YuNet·AuraFace처럼 `model_source.py` 경로 편입 가능.
