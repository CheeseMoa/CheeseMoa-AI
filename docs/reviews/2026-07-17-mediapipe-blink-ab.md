# 눈감음 판정 A/B 실측 — 현행 CNN vs MediaPipe Face Landmarker blendshape (2026-07-17)

## 배경과 방법

눈감음 CNN(open-closed-eye-0001)의 남은 실패는 도메인 문제다(운전자 모니터링 학습 —
유아·보정 이미지 오판, [원본 해상도 crop 기각 리뷰](2026-07-17-native-eyecrop-verdict.md)).
상용 가능(Apache 2.0) 대체 후보인 MediaPipe Face Landmarker(v2, blendshape 출력)를 실측 비교했다 —
`eyeBlinkLeft/Right` 점수를 현행 CNN의 양눈 closed 확률과 같은 얼굴에서 나란히 측정.

- 코퍼스: 실 이벤트 + 로컬 라벨셋 + event 61 = YuNet 검출 871 얼굴.
- 정합: YuNet bbox ↔ MediaPipe 랜드마크 bbox IoU 매칭. ADR-019 자격 얼굴 764개 중 585개(77%) 매칭.
- 판정 아날로그: 양눈 AND 규칙 = `min(blinkL, blinkR) ≥ t`.

### 임계 스윕 (자격+매칭 585 얼굴)

t=0.35~0.40에서 flagged 11 = 알려진 진짜 감음 전부(중복 포함), **오탐 0**. 분포: 감음
0.42~0.75 vs 뜬 눈 p90 0.138 — 경계 빈 구간 [0.36, 0.42]로 t=0.40이 안전. 현행 CNN처럼
극단적 양극화는 아니라 임계는 설정값으로 노출해야 한다.

## 판정 — 유망, 채택 전 잔여 검증 3건

blendshape 판정이 **알려진 실패 3종(유아 오탐·보정 이미지 미탐·수면 미탐)을 전부 해결**하면서
기존 정탐을 유지하고 585 얼굴에서 오탐 0이다. 얼굴 전체 컨텍스트를 보는 모델이라 눈 패치
CNN의 도메인 문제가 구조적으로 사라진다. 단:

1. **검출 커버리지 23% 공백**: MediaPipe 검출기는 극단적 회전·옆얼굴(45°+ 기운 초근접,
   누운 90° 옆얼굴 — 육안 확인)을 못 찾는다. YuNet+Umeyama 정렬은 롤을 보정하므로 이 얼굴들은
   현행 CNN만 판정 가능 → 교체가 아니라 **하이브리드**(MediaPipe 우선, 미검출 시 현행 CNN 폴백
   또는 미판정)로 설계해야 한다. 미검출분에 수면 감음 사진 1종이 실제로 있었다.
2. **arm64 검증**: EC2(t4g)용 aarch64 리눅스 휠 동작 확인 필요.
3. **메모리 실측**: mediapipe는 TFLite 런타임을 동반한다 — Spring과 공유하는 t4g.small(RAM
   1846MB)에서 상주 증가량 실측 후 채택 결정. 대안으로 .task 내부 모델의 ONNX 변환(기존
   onnxruntime 스택 유지) 검토 가능하나 변환 검증 비용이 붙는다.

도구: `scripts/`(gitignore) 외 스크래치패드 `mp_blink_survey.py`·`ab_analyze2.py`,
모델 `face_landmarker.task`(float16, 3.7MB, Apache 2.0).
