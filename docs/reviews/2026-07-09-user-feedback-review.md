## 동작 개요 (요약)

사용자가 앱에서 앨범을 수동 수정하면 Spring이 `cluster_feedback` 메시지를 발행하고, 워커는
이를 **재군집이 뒤집을 수 없는 제약(must-link/cannot-link)** 으로 번역해 영구 저장한다:

1. **스키마 검증** — action별 판별 유니온. 자기 병합·그룹 겹침·no-op 이동·중복 cluster_ids는
   파싱 시점에 거부(poison 즉시 격리).
2. **번역** (`_translate_feedback`) — image_id·cluster_id 세계를 face_id 쌍 세계로:
   - `merge`: target+source 전 멤버를 체인 must-link — 전이성으로 한 컴포넌트.
   - `split`: 그룹 내부 체인 must-link + 그룹 대표 간 cannot-link.
   - `reassign`: 이동 얼굴 ↔ 목적지 대표 must-link, ↔ 출처 잔류 대표 cannot-link.
     이동 얼굴의 기존 must-link는 전부 폐기(옛 동료를 끌고 가는 것 방지).
     `from_cluster_id == "__uncertain__"`이면 미배정(cluster_id=None) 얼굴이 대상.
   - `confirm_distinct`: 각 클러스터 대표 얼굴 전 쌍 cannot-link 클리크(merge의 반대 선언).
   - 스테일 참조(사라진 cluster_id·사진)는 경고 후 건너뛰고, 빈 번역이어도 진행.
3. **later-wins 조정** (`_reconcile_constraints`) — 새 쌍은 무조건 수용, 기존 쌍은 최신부터
   역순으로 걸러 나중 결정과 충돌하는 오래된 결정이 먼저 탈락. must/cannot 동률은 must 우선.
4. **재군집 + 저장** — 조정된 제약 셋으로 event 전체 재군집(제약이 사람 결정을 강제), `.npz`에
   새 배정과 제약을 함께 저장, 전체 스냅샷을 결과 큐로 발행.

핵심 설계: 보정은 "결과를 직접 고치는 패치"가 아니라 **재군집의 불변 조건**이다. 그래서 이후
어떤 사진이 추가·삭제돼 재군집이 반복되어도 사람의 결정이 유지된다 — 단, 아래 1번의 예외가 있다.

## 운영 환경 리스크

### 1. 보정의 내구성이 앵커 얼굴의 생존에 의존 — 삭제·후속 보정이 과거 결정을 조용히 무효화

앨범 수준의 사용자 결정("A와 B는 다른 사람", "이 사진은 B 사람")이 전부 **특정 얼굴 쌍**으로
인코딩되는데, 그 얼굴이 사라지면 결정도 함께 사라지고 아무도 통보받지 못한다:

- **confirm_distinct**: 대표 얼굴 쌍에만 cannot-link를 건다. 대표 얼굴의 사진이 삭제되면
  `masked_by_photo_ids`가 댕글링 쌍을 프루닝하고(구조상 필연 — 남기면 로드 불변식이 깨짐),
  "두 앨범은 다른 사람" 확정이 소멸한다. 이후 다리 사진이 들어오면 confirm_distinct가 막으려던
  오병합이 그대로 재발한다.
- **merge**: 체인 (f0,f1),(f1,f2),… 인코딩이라 **중간 얼굴 하나만 삭제돼도 체인이 두 컴포넌트로
  단절**된다 — 병합 결정이 부분 상실되고, 유사도가 애매한 경우 앨범이 도로 갈라질 수 있다.
- **reassign**: "사진 X는 B 사람" 결정이 (X 얼굴, B 대표 b0) 쌍으로 저장되는데, 이후 b0의 사진을
  다른 곳으로 reassign하면 superseded 규칙이 b0의 기존 must-link를 전부 폐기하므로 **X의 편입
  결정까지 함께 소실**된다. X는 다음 재군집에서 기하(유사도)대로 되돌아갈 수 있다.

개별 규칙은 각자 정당하다(댕글링 프루닝은 필연, superseded 폐기는 옛 동료 견인 방지). 문제는
합성 효과 — "재군집이 사람 결정을 뒤집지 않는다"는 핵심 계약이 삭제·후속 보정 시나리오에서
구멍이 나는데, 로그조차 남지 않는다.

- 권장: ① 최소 조치 — `masked_by_photo_ids` 프루닝과 superseded 폐기가 실제로 쌍을 제거할 때
  개수·종류를 warning 로그로 표면화(운영에서 "보정했는데 풀렸어요" 문의의 유일한 단서),
  ② 구조 조치 — 프루닝·폐기 시 같은 클러스터의 다른 생존 얼굴로 쌍을 **재앵커링**(체인은 재연결,
  대표는 차순위 얼굴로 교체)하면 결정의 의미가 보존된다. merge를 체인 대신 star(허브 1 + N)로
  인코딩해도 중간 단절은 사라지지만 허브 삭제 취약성은 남으므로 재앵커링이 더 근본적,
  ③ Spring과 정책 합의 — 재앵커링 불가능(클러스터 전멸) 시 결정 소멸을 결과로 통보할지

### 2. 무효(스테일) 보정이 `succeeded`로 보고됨 — Spring이 반영 여부를 구분할 수 없음

스테일 참조로 번역이 비어도(merge 대상 전부 소멸, reassign 대상 얼굴 없음 등) 경고 로그만 남기고
재군집·스냅샷 발행을 진행하며 결과는 `succeeded`다. 스냅샷 발행 자체는 옳은 설계지만(Spring이
현재 상태로 수렴), **결과 계약에 "보정이 적용되었는가" 신호가 없어** 사용자 관점에서는 "병합
버튼을 눌렀는데 아무 일도 없고 오류도 없는" UX가 된다. 보정과 classify가 경합하는 타이밍
(사용자가 오래된 화면에서 조작)에서 실제로 발생할 수 있는 시나리오다.

- 권장: Spring과 결과 계약에 적용 여부 필드(예: `feedback_applied: bool` 또는 무시 사유) 추가를
  협의 — 로드맵의 "Spring 실계약 통합검증"·"confirm_distinct 트리거 정책 합의" 항목과 같은
  협의 트랙이다. 워커 단독으로는 현 상태가 최선이므로 계약 확장 전까지는 경고 로그가 유일한
  관측 수단임을 인지할 것

### 3. `reassign.to_cluster_id = "__uncertain__"` — 미정의 의미론이 조용히 통과

예약 앨범 id는 `from_cluster_id` 쪽만 계약이다(uncertain 편입). 그런데 `to_cluster_id`에 예약
id가 오면 스키마가 거부하지 않고, 번역에서 "목적지 클러스터가 비어 있음" 경고와 함께 must-link
없이 출처 잔류 대표와의 cannot-link만 생성된다 — 우연히 "이 사진은 이 사람이 아님(소속 해제)"
유사 동작이 성립한다. 유용해 보이는 동작이지만 **계약에 없는 우연**이라, 향후 번역 로직이
바뀌면 Spring 쪽 의존이 소리 없이 깨질 수 있다.

- 권장: 스키마에서 `to_cluster_id == UNCERTAIN_ALBUM_ID`를 명시 거부하거나, 반대로 "소속 해제"
  액션으로 공식 승격해 feature-spec §6.3에 계약으로 기록 — 어느 쪽이든 현재의 미정의 상태만
  해소하면 된다

## 유지보수성

### 4. 소소한 것들

- **다중 미배정 얼굴 사진의 `__uncertain__` reassign**: 미배정 얼굴이 2개 이상인 사진(낯선 2인
  단체 — 계약상 uncertain이 아니라 common으로 라우팅됨)에 Spring이 계약을 어기고 uncertain
  reassign을 보내면, 서로 다른 두 타인이 같은 목적지 대표와 must-link로 묶여 **전이적으로 동일
  인물이 된다**. 계약 위반 경로라 우선순위는 낮지만, `moved`가 2개 이상 + `from_uncertain`일 때
  경고 하나면 오용을 조기에 잡는다.
- **보정 경고 로그의 event_id 누락**: `_translate_feedback`의 warning들이 job_id는 찍지만
  event_id가 없는 메시지가 있다(merge/split/reassign). 스테일 보정 조사는 event 단위로 하므로
  event_id를 일관되게 포함할 것 — 1·2번의 관측성 조치와 같은 계열.

## 확인해서 문제 없었던 것 (오탐 방지 기록)

- **`_reconcile_constraints`의 later-wins 정확성**: 체인 충돌 케이스(cannot (x,y) vs 옛 must
  (x,m),(m,y))를 수동 추적 검증 — 최신 must부터 역순 수용 + `would_connect`의 루트 쌍 비교
  `{root_x, root_y} == {find(a), find(b)}`가 "이 must를 살리면 새 cannot이 깨지는가"를 정확히
  판정하고, 오래된 쪽만 탈락한다. must/cannot 동률에서 must 우선은 임의 규칙이 아니라 "종류별
  배열이라 상대 시간순이 없다"는 저장 구조의 한계를 문서화한 명시적 결정이다.
- **스키마 계약 검증의 완결성**: 자기 병합(target ∈ sources)·split 그룹 겹침(must와 cannot 동시
  → 재군집의 난해한 인덱스 모순으로 번지기 전에 원인 그대로 거부)·no-op 이동(from == to)·
  confirm_distinct 중복 id가 전부 파싱 시점에 거부된다. 잘못된 조합(merge인데 split payload)은
  판별 유니온이 구조적으로 차단 — 검증이 경계에 정확히 배치되어 있다.
- **유령 event 보정의 FIFO 비차단**: 저장된 적 없는 event의 보정은 재시도로 해소되지 않는
  결정적 이상이므로 `failed` 반환(발행 후 삭제)으로 FIFO 그룹이 막히지 않게 한다 — delete의
  멱등 succeeded와의 구분(지울 게 없음 vs 보정할 대상이 없음)이 의도적이고 올바르다.
- **대표 얼굴 선정의 결정성**: 행 순서상 첫 얼굴 고정 — .npz 행 순서는 append-only라 메시지
  재전달 시에도 같은 대표가 선정된다(멱등 재처리 안전성의 근거). 대표 하나로 충분한 이유
  (`_enforce_cannot_link`가 위반 라벨을 쪼갠 뒤 나머지를 최근접 앵커로 재배정)도 주석에 명시.
- **reassign superseded 규칙의 근거**: 이동 얼굴의 기존 must-link 전부 폐기는 "체인 구조상
  어느 쌍이 낡은 결정인지 특정 불가 → later-wins에 맡기면 옛 동료를 새 인물로 견인"이라는
  실패 시나리오와 함께 문서화되어 있다(1번의 부작용은 이 규칙 자체가 아니라 합성 효과의 문제).
- **자가검증 커버리지**: 병합→분리 왕복의 later-wins(④), 기하를 이기는 사용자 결정(⑤), 제약
  당사자 삭제 시 프루닝(⑥), uncertain 편입(⑫), confirm_distinct 오병합 차단(⑬ — 실측 오병합
  기하로 회귀 고정)까지 보정 경로의 핵심 시나리오가 `__main__` 자가검증에 고정되어 있다.
  pytest 승격(로드맵) 시 그대로 옮기면 된다.

## 우선순위
