# 입력 품질이 임베딩 판별력보다 먼저다 — 정렬 에일리어싱 + 랜드마크 지터 (2026-07-14)

test4(인물 2명, 9장) 성능 테스트에서 앨범이 3개 생긴 것을 추적하다가, **파이프라인이 스스로 주입하는
노이즈가 우리가 측정하려는 신원 신호보다 크다**는 것을 확인했다. 모델 교체 검토보다 앞서야 하는 문제다.

## 발단 — test4 재현

동일 인물 B가 앨범 2개로 파편화(001·002 / 003~006). 파편 간 centroid 유사도 **0.587**로 병합 임계
0.68에 미달. 처음엔 초근접 왜곡·표정 변화(임베딩 모델 한계)로 판단했으나, 입력 단계를 파보니
그보다 **큰 변수 두 개**가 먼저 있었다.

## 발견 1 — 정렬 안티에일리어싱 누락 (코드 결함)

`app/pipeline/align.py`의 `cv2.warpAffine(image, M, (112, 112))`는 **기본 보간 INTER_LINEAR**를 쓴다.
INTER_LINEAR는 2×2 이웃만 샘플링하므로, 큰 축소에서 저역통과 없이 원본을 서브샘플링한다 — 교과서적
에일리어싱이다.

- test4 실측 축소 배율: 평균 **6.0배**, 최대 **12.7배** (원본 얼굴 900px → 112px)
- 12배 축소면 12×12=144픽셀 블록에서 4픽셀만 읽고 140개를 버린다
- 어느 픽셀이 걸리는지가 서브픽셀 위치에 좌우돼, **같은 얼굴도 촬영마다 크롭의 고주파 패턴이 달라진다**

축소 배율에 맞춘 가우시안 프리블러(σ ≈ (1/scale)/2)를 넣고 재측정:

| 지표 | 현행 | 안티에일리어싱 |
|---|---|---|
| B 파편 간 centroid | 0.587 | **0.640** (+0.053) |
| B 파편 간 쌍 평균 | 0.486 | **0.541** |
| A 내부(동일인) 쌍 평균 | 0.632 | **0.654** |
| A↔B 타인 max | 0.275 | 0.278 (거의 불변 — 판별력이 실제로 오른 것) |

비용은 블러 1회. `warpAffine`은 INTER_AREA를 지원하지 않으므로 **프리블러 방식이 맞다**.

## 발견 2 — 랜드마크 지터가 임베딩보다 큰 변수 (구조적)

**같은 사진, 같은 얼굴인데 검출 해상도만 바꾸면 임베딩이 크게 흔들린다.**
`max_side`를 1600/2000/2400(전부 정상 동작 범위)으로 바꿔 같은 얼굴을 검출·임베딩한 결과:

| 정렬 방식 | 같은 얼굴 임베딩 유사도 (1.0이 완벽) |
|---|---|
| 현행 (AA 없음) | 평균 0.8895, **최저 0.4288** |
| AA 정렬 | 평균 0.9147, 최저 0.4531 |
| AA + 2단계 랜드마크 정제 | 평균 0.9222, **최저 0.6290** |

**노이즈 바닥(0.43)이 인물 B 파편 간 거리(0.587)보다 크다.** 임베딩이 신원이 아니라 랜드마크 지터를
상당 부분 측정하고 있다는 뜻이다.

원인: **YuNet은 WIDER FACE(작은 얼굴 위주)로 학습돼 초근접 대형 얼굴이 학습 분포 밖**이다.
- 랜드마크 이동량: 얼굴폭 대비 평균 **3.17%**, 최대 **15.78%**
- 원본 해상도(max_side=None) 검출은 오검출이 폭증(28개 → 85개)해 더 나쁘다 — `max_side=2000`을 올리는
  방향은 해법이 아니다
- 2단계 정제(bbox 여유 크롭 → 얼굴폭을 정규 스케일 160px로 리사이즈 → 랜드마크 재추출)로 최저값이
  0.45 → 0.63으로 개선. 지터를 실제로 줄이는 방향은 맞으나, test4 최종 군집은 못 바꿨다
  (centroid 0.61) — **지터 억제와 세션 간 파편 접합은 별개 축**이다. 파라미터(정규 스케일·마진) 스윕 필요.

## 발견 3 — 참고: 임베딩 모델 격차 (라이선스로 막힘)

동일 crop에 AdaFace IR-101(WebFace12M)을 꽂으면 파편 간 centroid 0.587 → **0.809**, 동일인−타인 간격이
+0.17 → **+0.42**로 벌어지고 현행 설정 그대로 앨범이 정확히 2개가 된다(노이즈 0). 속도도 동일
(2스레드 65ms/얼굴). **그러나 상용 불가** — WebFace260M 계열 학습 데이터가 "데이터셋과 그 서브셋의
상용 이용 금지"이고, AdaFace의 MIT는 코드 전용이다. 상세: 아래 라이선스 절.

TTA(좌우 반전 평균)는 **기각**. 파편 간 centroid가 0.587 → 0.642로 오르지만 타인 유사도도 함께 올라
판별력 이득이 없고(동일인−타인 간격 불변), 임베딩 비용은 2배, test4에서는 군집이 오히려 3조각으로
악화됐다. ArcFace 계열은 학습 시 flip augmentation을 이미 쓰므로 반전 임베딩이 원본과 거의 같은 방향이다.

## 라이선스 조사 결과 (2026-07-14, deep-research)

**무료 + 상용 가능 + AuraFace보다 판별력 우수한 모델은 존재하지 않는다.** 라이선스는 코드가 아니라
**가중치·학습 데이터**에서 막힌다:

| 모델 | 코드 | 가중치/데이터 | 판정 |
|---|---|---|---|
| InsightFace buffalo_l·antelopev2 | MIT | model_zoo README: "ALL models are available for non-commercial research purposes only" | 상용 라이선스 **구매 가능**(recognition-oss-pack@insightface.ai) |
| AdaFace 전 학습본 | MIT (코드만) | HF 모델 카드에 license 필드 없음, "학습 데이터셋 라이선스를 따르라" | 상용 불가 |
| Glint360K 파생 전부 | — | "데이터셋 **및 그 데이터로 학습된 모델**은 비상용 연구 전용" | 상용 불가 |
| WebFace260M 계열(600K/12M/4M) | — | "이 데이터셋 **과 그 서브셋**의 상용 이용 금지" | 상용 불가 |
| ByteDance LVFace | HF 태그 `mit` | 본문: "모델은 비상용 연구 목적만", Glint360K 학습 | 상용 불가 (**태그가 함정**) |
| OpenCV SFace | Apache-2.0 | 학습 데이터 미공개(MS1M 계열 추정), MobileFaceNet급·128차원 | AuraFace보다 **약함** |
| 합성 데이터 학습 | — | 최고(Arc2Face) IJB-C 92.66 vs 실데이터 96.67. DigiFace-1M은 R-UDA로 "학습된 모델도 상용 금지" | 정확도·라이선스 양면 미달 |

**AuraFace(fal.ai, Apache-2.0)가 사실상 유일한 합법 선택지**다 — R100급·512차원·ArcFace 정렬 호환
임베더 중 유일하게 LICENSE.md에 Apache-2.0이 커밋돼 있다. 다만 학습 데이터는 "여러 출처의 상용
데이터셋"이라고만 적혀 있어 감사 가능한 출처 사슬은 없다.

buffalo_l 상용 라이선스를 검토한다면 **선행 조건 2개**:
1. 공개 코호트 수치가 **동아시아 74.96 vs 백인 94.70** — 한국인 사용자 기반에서 이득이 실현되는지
   자체 데이터로 A/B 검증 후에 결제할 것
2. 유료 라이선스가 InsightFace 모델 IP만 커버하는지, 그 밑의 Glint360K/WebFace600K **데이터 출처까지
   면책**하는지 서면 확인

자체 학습(AdaFace 손실 + 자체 데이터)은 비추천: WebFace4M(400만 장/20만 명)이 실질 하한선인데 우리
데이터는 6자리수 부족하고, **fal.ai가 정확히 이 길을 가서 나온 결과물이 지금 우리가 쓰는 AuraFace**다
(본인들이 "데이터 규모 부족"을 자인, CFP-FP 3.7pp 미달). 사용자 사진 학습은 개인정보보호법상 얼굴이
민감정보(생체인식정보)라 별도 명시 동의 + 목적 외 이용 문제가 걸린다.

## 결론 — 작업 순서 원칙

**입력 품질(정렬·랜드마크)이 임베딩 모델 선택보다 먼저다.** 노이즈 바닥이 0.43인 파이프라인에 더 좋은
임베딩 모델을 얹는 것은 밑 빠진 독에 물 붓기이며, AdaFace가 잘 나온 것도 상당 부분 "지터에 더
강인해서"일 수 있다 — 즉 유료 라이선스로 살 이득의 일부를 무료로 회수할 여지가 있다.

1. **정렬 안티에일리어싱 수정** — 작고 확실함(+0.053), 비용 사실상 0
2. **랜드마크 지터 정면 대응** — 2단계 정제 파라미터 스윕(정규 스케일·마진), 또는 큰 얼굴 전용 랜드마크
   경로. `max_side`를 올리는 방향은 오검출 폭증으로 **해법이 아님**
3. **그 다음에** 남은 격차를 보고 모델 교체/유료 라이선스를 판단

## 미해결 — 원인 규명 선행 과제

test4는 인물 2명짜리 **n=1 샘플**이다. 라벨된 자체 데이터로 **동일 인물 세션 간 코사인 거리 분포 vs
타인 간 분포**를 그려, 겹침이 실제로 병합 임계(0.68)를 침범하는지 확인해야 한다. 겹치면 임베딩이
원인이므로 유료 라이선스가 정당화되고, 안 겹치면 버그는 `cluster.py`에 있어 모델 교체는 돈 낭비다.
(deep-research 검증자들도 "CFP-FP/CPLFW는 검증(verification) 지표이지 군집 응집도가 아니다"라며
임베딩 품질 → 파편화의 인과를 "정황 일치, 미입증"으로 판정했다.)

## 구현 결과 (2026-07-15 후속)

두 처방을 구현·배포 준비 완료 (`feature/clustering-fragmentation-fix`, 정렬 AA + 랜드마크 2단계 정제 커밋).
측정은 재작성한 `scripts/measure_landmark_jitter.py`(로컬 전용)로 수행 — 대상을 test2~4 전체(30장,
매칭 얼굴 39개)로 넓혀 리뷰 당시(test4 위주)보다 노이즈 바닥이 더 나쁜 **0.3254**까지 관측됐다
(리뷰의 0.4288은 두 번째 최악 얼굴로 정확히 재현).

| 지표 (같은 얼굴, max_side 1600/2000/2400 스윕) | 현행 | AA | AA+정제(최종) |
|---|---|---|---|
| 유사도 평균 | 0.9133 | 0.9287 | **0.9596** |
| 유사도 최저 | 0.3254 | 0.3400 | **0.6881** |
| 랜드마크 이동(얼굴폭 %) 평균/최대 | 5.06 / 26.52 | (동일) | **3.07 / 11.01** |

- **정제 파라미터는 그리드 스윕으로 리뷰 초기값(160/0.5)이 아닌 224/0.75 확정** — 정규 폭
  {128,160,192,224} × 마진 {0.3,0.5,0.75} 12조합 중 전 지표 최고. 160/0.5는 최저 0.4369에 그쳤다.
  (마진이 클수록 크롭에 문맥이 넉넉해 YuNet 재검출이 안정되는 경향, 단 128/0.75·192/0.3 같은
  조합은 오히려 폭주 사례가 있어 개별 축 단조 아님 — 스윕 없이 직관으로 정할 수 없는 값이었다.)
- 정렬 AA는 ROI 한정 프리블러로 구현(전체 이미지 블러와 픽셀 동일함을 동등성 검증 ⑤로 고정).
  확대(s≥1)·`antialias=False` 경로는 기존과 픽셀 단위 동일(검증 ④). insightface 파리티 검증
  1~3항도 그대로 통과.
- 검출 비용 +34%(로컬 실측 2237→2988ms/90회) — 파이프라인 병목은 임베딩(EC2 476ms/얼굴)이라
  전체 비용 영향은 수 % 미만. 토글(`DETECT_REFINE_LANDMARKS` 등 5종)로 .env 롤백 가능.
- **test4 군집은 리뷰의 예고대로 안 바뀌었다** — B 파편 간 centroid는 0.5867(리뷰 0.587 재현)에서
  교정 후에도 0.51~0.59로 병합 임계 0.68 미달. 파편 수는 2→3이 됐으나(AA 단독에서도 동일 —
  크롭이 바뀌며 HDBSCAN 절단면이 달라진 것) B 커버리지는 6→7장으로 늘었다(007이 앨범 편입).
  **지터 억제와 세션 간 파편 접합은 별개 축**이라는 리뷰 결론 그대로이며, 다음 단계는 아래
  "미해결 — 원인 규명 선행 과제"다.

## 재현 방법

`docs/reviews/` 관례상 스크립트는 남기지 않았다. 재현 요지:
- 정렬 AA: `align_face`에서 `M`의 det로 등방 스케일 s를 뽑아 `s < 1`이면 σ=(1/s)/2 가우시안 블러 후 warp
- 지터 측정: `DetectorConfig(max_side=...)`를 1600/2000/2400으로 바꿔 같은 얼굴(bbox IoU>0.5로 매칭)을
  임베딩해 코사인 유사도 비교
- 2단계 정제: bbox를 max(w,h)*0.5 마진으로 크롭 → 얼굴폭 160px가 되도록 (AA 후) 리사이즈 → 재검출로
  랜드마크 획득 → 원본 좌표계로 역변환
