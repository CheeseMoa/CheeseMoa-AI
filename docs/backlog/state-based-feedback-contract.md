# 제안: 상태 기반 보정 계약 — 검토완료 앨범 멤버십을 진실로 (ADR-010 후보)

  `app/schemas/messages.py`, `app/handlers.py`, Spring 발행부
- 배경: [user-feedback-constraints.md](../architecture/user-feedback-constraints.md)(현행 액션 기반 설계)

## 문제 제기

`.npz`가 사용자 보정(must-link/cannot-link)을 들고 있는데, **Spring의 PostgreSQL에도 "어떤 사진이
어떤 앨범에 속하는가"가 이미 저장돼 있다.** 사용자 결정이 두 곳에 살고 있고, 실제로 오늘
[불일치 사고](../reviews/2026-07-11-duplicate-embedding-split.md)가 났다(삭제가 한쪽에만 반영).
워커가 제약을 따로 보관하지 않고 Spring이 멤버십을 보내주면 되지 않는가?

### 단순 멤버십만으로는 안 된다 (기각된 형태)

Spring의 앨범 멤버십은 **결과**이고 워커가 필요한 건 **의도**다. 앨범 A에 사진 5장이 있다는
사실만으로는 *사용자가 묶은 것*인지 *알고리즘이 묶은 것*인지 알 수 없다.

- 멤버십을 그대로 must-link로 바꾸면 **알고리즘의 자동 판단이 전부 사용자 결정으로 승격되어 동결**된다
  → 새 사진이 들어와도 재군집이 과거 오류를 못 고친다. "전체 재군집이 군집의 진실"([ADR 003](../decisions/003-full-reclustering.md))이 무너진다.
- "현재 다른 앨범 = cannot-link"도 마찬가지다. **알고리즘이 나눈 것과 사용자가 다른 사람이라고
  확정한 것을 구분하는 것이 `confirm_distinct`의 존재 이유** 그 자체다.

## 핵심 관찰 — 검토(review) 상태가 그 구분을 담고 있다

앱에는 앨범 단위 **미검토 / 검토완료** 상태가 있다. 검토완료 = 사용자가 그 앨범의 멤버십을
확인했다는 뜻이므로, **검토완료 앨범의 멤버십은 결과가 아니라 의도다.**

이 한 가지 정보가 위 기각 사유를 무력화한다.

## 제안

```
검토완료 앨범 내부  → must-link
검토완료 앨범 사이  → cannot-link (대표 얼굴 간)
미검토 앨범        → 제약 없음 (알고리즘 산출물 그대로, 지금과 동일)
```

이 두 줄로 현행 액션 4종의 의미가 전부 재현된다:

| 사용자 행동 | 현행 (액션 기반, 명령형) | 제안 (상태 기반, 선언형) |
|---|---|---|
| 앨범 A·B 합침 | `merge` → must-link 체인 | A∪B가 한 앨범으로 검토완료 |
| 앨범 쪼갬 | `split` → must-link + 그룹 간 cannot-link | 두 앨범이 각각 검토완료 |
| 사진 옮김 | `reassign` → must-link + 출처 cannot-link | 그 사진이 B 멤버십으로 검토완료 |
| 다른 사람 확정 | `confirm_distinct` → 대표 간 cannot-link | A·B 둘 다 검토완료 = 서로 다름 |

## 이득

### 1. 충돌 해소 로직이 통째로 사라진다

매 요청이 **전체 상태**를 싣고 오므로 "과거 결정과 새 결정이 모순되면?"이라는 문제가 발생하지 않는다.
`handlers.py`에서 다음이 삭제 대상이 된다:

- `_reconcile_constraints` (later-wins 조정)
- `_FaceUnionFind` (`would_connect` 판정)
- `_translate_feedback`의 4갈래 분기
- 스테일 참조 처리, `superseded_faces` 프루닝

[reassign 동률 결함](../reviews/2026-07-10-reassign-mustlink-tiebreak.md)(36627d6)은 이 모델에선
**발생 자체를 하지 않는다** — 이동 방향을 cannot-link로 우회 표현할 필요 없이 "이 사진은 B 소속"이라고
직접 선언하기 때문.

### 2. 사용자 결정의 단일 진실이 Spring이 된다

두 저장소가 어긋날 여지가 없어진다. 오늘 사고의 근본 구조가 사라진다.

### 3. `confirm_distinct` 트리거 정책 미결이 자동 해소된다

멤버십 선언이 곧 이격 선언이므로 별도 트리거 시점 합의가 필요 없다.

## 선결 조건 / 위험

### ① "검토완료"의 의미가 강한 보증인가 — **제품 결정, 가장 중요**

- **강한 보증** ("이 앨범의 *모든* 사진이 이 사람이 맞다") → must-link 동결 정당
- **약한 확인** ("한 번 훑어봤다") → **대충 누른 검토완료가 오류를 영구 고착시킨다**

실제 사례: 오늘 잘못 쪼개진 앨범 중 하나가 이미 **검토완료 상태**였다. 상태 기반 모델이었다면
그 잘못된 분리가 must-link로 굳었을 것이다. (사용자가 다시 합치면 풀리지만, "검토완료를 누르는
순간 동결된다"는 것을 UI가 사용자에게 알려야 한다.)

**앨범에 사진 200장이 있는데 사용자가 대충 훑고 검토완료를 눌렀다면 200장이 전부 must-link로
묶인다** — 이 리스크를 감당할 수 있는지가 채택 여부를 가른다.

### ② 얼굴 단위 매핑은 `.npz`에 계속 있어야 한다

Spring의 계약은 전부 `image_id` 단위인데(`ClassifyRequest.images`, `ReassignPayload.image_id`) 워커는
**face 단위**로 동작한다. 단체 사진 P의 얼굴 2개가 앨범 A·B에 나뉘어 들어간 경우, **P의 어느 얼굴이
A인지 Spring은 모른다.**

→ `face_ids` / `photo_ids` / `cluster_ids` 배열은 워커가 계속 보관한다. 어차피 임베딩 때문에 `.npz`는
존재하고 이 배열들은 수 KB다. **없어지는 것은 `must_link_pairs` / `cannot_link_pairs` 두 배열뿐이며,
파일 용량은 사실상 그대로다** (저장 비용 절감은 채택 근거가 못 된다).

### ③ 워커의 자족성을 포기한다

현재 워커는 `event_id` 하나로 완결된다. 제안 후에는 **Spring이 매 요청마다 멤버십을 정확히 실어
보내야만** 동작한다. Spring 쪽 누락 한 번이 **사용자가 며칠에 걸쳐 고친 결정을 조용히 날린다** —
오늘 `delete_request` 미도달 사고와 같은 계열의 위험이다.

### ④ 메시지 크기

`classify_request`에 event 전체 멤버십(수백 사진 × 앨범 id + 검토 플래그)이 실린다. 수 KB 수준이라
SQS 256KB 한도에는 여유가 있으나, 대형 이벤트에서 재확인 필요.

## 마이그레이션

기존 `.npz`의 제약을 버리고 Spring의 검토완료 멤버십으로 갈아타는 절차가 필요하다. 기존 제약 →
멤버십 역변환은 불가능하지 않다(현재 `cluster_ids`가 이미 보정 반영 결과이므로), 다만 어떤 앨범을
"검토완료"로 간주할지는 Spring DB의 검토 상태에 의존한다.

## 판단

**방향은 타당하고 장기적으로 더 나은 설계다.** 검토 단계가 있다는 전제 위에서 상태 기반이 액션
기반보다 단순하고, 두 저장소의 불일치 위험도 사라진다.

다만 계약 전면 개편이고 **①이 제품 결정**이라 팀 합의가 선행돼야 한다. 착수 순서:

1. P0 데이터 오염 해소 (데이터를 먼저 안정시킨다)
2. ①에 대한 제품 합의 → 이 문서를 ADR-010으로 승격, 메시지 스키마·마이그레이션 확정
3. 구현 (`_reconcile_constraints` 계열 삭제로 코드는 오히려 줄어든다)
